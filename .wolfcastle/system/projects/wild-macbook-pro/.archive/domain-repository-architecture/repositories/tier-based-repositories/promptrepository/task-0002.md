# Implement PromptRepository and tests

Implement PromptRepository in internal/pipeline/repository.go following the contract spec written in the previous task.

Key implementation details:
- The struct holds a single tiers (tierfs.Resolver) field
- NewPromptRepository(wolfcastleRoot) creates tierfs.New(filepath.Join(wolfcastleRoot, "system")) and delegates to NewPromptRepositoryWithTiers
- Resolve(name, ctx): calls tiers.Resolve("prompts/"+name+".md") to get raw bytes, converts to string. If ctx is nil, returns raw. Otherwise parses with text/template and executes with ctx. Returns result string
- ResolveRaw(category, name): calls tiers.Resolve(category+"/"+name) and returns string(data)
- ListFragments(category, include, exclude): calls tiers.ResolveAll(category) to get map[string][]byte. Applies include/exclude filtering (same logic as existing ResolveAllFragments). Returns content strings sorted by filename
- WriteBase delegates to tiers.WriteBase
- WriteAllBase walks an fs.FS using fs.WalkDir, reads each file, calls tiers.WriteBase for each

Tests (repository_test.go) should use testutil.NewEnvironment(t) and exercise:
- Resolve returns raw content when ctx is nil
- Resolve executes template when ctx is provided
- Resolve returns error for missing prompt
- Resolve returns error for invalid template syntax
- ResolveRaw returns content from highest-priority tier
- ResolveRaw with tier override (custom overrides base, local overrides custom)
- ListFragments returns all .md files sorted by name
- ListFragments applies include filtering
- ListFragments applies exclude filtering  
- ListFragments with tier override semantics (higher tier replaces same-named file)
- WriteBase persists and is reflected in subsequent Resolve
- WriteAllBase writes multiple files from fs.FS

Do NOT modify the existing fragments.go functions or the Tiers variable. The repository is a parallel implementation using tierfs.Resolver. Cleanup happens later.
