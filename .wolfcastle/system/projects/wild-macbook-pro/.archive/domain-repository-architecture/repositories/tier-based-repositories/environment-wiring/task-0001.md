# Wire ConfigRepository and PromptRepository into testutil.Environment

Uncomment the Config and Prompts fields on the Environment struct and populate them in NewEnvironment.

Changes to internal/testutil/environment.go:
1. Add imports for the pipeline package (config is already imported)
2. Uncomment the Config field: Config *config.ConfigRepository
3. Uncomment the Prompts field: Prompts *pipeline.PromptRepository
4. In NewEnvironment, after constructing the tierfs.FS (the tiers variable), construct both repositories using their WithTiers constructors:
   - config.NewConfigRepositoryWithTiers(tiers, wolfcastleDir) for Config
   - pipeline.NewPromptRepositoryWithTiers(tiers) for Prompts
5. Set both on the returned Environment struct

The key insight: NewEnvironment already creates a tierfs.FS (the tiers variable on line 96). The repositories share this same resolver instance, which means tests that write to tier directories via tiers.WriteBase will be reflected in subsequent repository reads. This is intentional — the WithTiers constructors exist precisely so that test infrastructure can inject a shared resolver.

Verify that existing tests still pass after wiring. The new fields are additive — nothing currently reads them, so existing tests should be unaffected. Run go test ./internal/testutil/... and go test ./internal/config/... and go test ./internal/pipeline/... to confirm.
