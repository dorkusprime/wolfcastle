# Write config write commands spec

Create a spec document via 'wolfcastle spec create' that defines all four config write commands: set, unset, append, remove.

The spec must cover:

1. CLI interface for each command:
   - 'wolfcastle config set <key> <value> [--tier local|custom]' — sets a config key using dot-notation. Value is parsed as JSON (bare strings allowed). Tier defaults to local.
   - 'wolfcastle config unset <key> [--tier local|custom]' — removes a key by writing null (deletion semantics via DeepMerge). Tier defaults to local.
   - 'wolfcastle config append <key> <value> [--tier local|custom]' — appends a value to an existing array. Tier defaults to local.
   - 'wolfcastle config remove <key> <value> [--tier local|custom]' — removes a value from an array. Tier defaults to local.

2. Dot-notation path semantics:
   - Paths like 'daemon.poll_interval_seconds' or 'pipeline.stages.audit.enabled' split on dots
   - No array indexing (e.g., 'commands[0]' is NOT supported)
   - Map keys within nested maps (like pipeline.stages, models) addressed by their key name
   - Intermediate paths that don't exist are created automatically for set

3. Value parsing:
   - Values are first tried as JSON (numbers, booleans, null, objects, arrays)
   - If JSON parse fails, the raw string is used as a JSON string value
   - This means 'wolfcastle config set logs.max_files 5' sets an integer, and 'wolfcastle config set identity.user alice' sets a string

4. Tier targeting:
   - --tier flag accepts 'local' or 'custom' only (writing to base is not allowed)
   - Default tier is 'local'
   - Note existing ADR: WithConfig writes to custom tier, but user-facing CLI defaults to local per scope definition

5. Read-modify-write flow:
   - Read the current tier overlay file (may be empty/missing)
   - Apply the mutation to the overlay map
   - Write the modified overlay back to the tier file
   - Load the fully merged config (all tiers + defaults) and run ValidateStructure
   - If validation fails, roll back the tier file and return the validation error
   - If validation passes, print a success message

6. Output:
   - Human mode: print 'Set <key> = <value> in <tier>/config.json' (or equivalent for other commands)
   - JSON mode: output.Ok envelope with the key, value, and tier

7. Error conditions:
   - Invalid tier name
   - Malformed dot-notation key (empty segments, trailing dots)
   - config append/remove on a non-array value
   - config remove for a value not present in the array
   - Validation failure after write (must roll back)
   - Cannot write to base tier

Reference existing patterns: cmd/config/show.go for CLI structure, readTierFile for tier reading, internal/config/repository.go for WriteCustom/WriteLocal, internal/config/merge.go for DeepMerge semantics.
