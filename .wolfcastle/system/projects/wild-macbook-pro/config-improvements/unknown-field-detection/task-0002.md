# Implement unknown field detection

Implement unknown-field detection in config unmarshalling per the spec written in task-0001.

Key implementation details:
- Modify internal/config/config.go Load() to detect unknown fields. The most natural insertion point is after the merged JSON is produced (line ~163 in current code) but before or during the final json.Unmarshal into Config.
- If using DisallowUnknownFields: replace the plain json.Unmarshal with json.NewDecoder + DisallowUnknownFields(). Catch the error, extract the field name, and convert to a warning rather than a hard failure.
- If using the diff approach: after unmarshalling, round-trip the Config struct back to map[string]any and compare keys recursively against the pre-unmarshal merged map.
- For per-tier detection: in the tier-loading loop, attempt a strict unmarshal of each tier's raw JSON against the Config struct to identify which tier introduced the unknown field. This gives better error messages.
- Add a Warnings []string field to Config, or return warnings as a separate value from Load(). Evaluate caller impact and choose the least disruptive approach.
- Surface warnings in the CLI: callers of app.Config.Load() (in cmd/root.go PersistentPreRunE and cmd/init.go) should print warnings to stderr.
- Write comprehensive tests covering: single unknown top-level field, unknown nested field, unknown field in a specific tier, multiple unknown fields, no unknown fields (clean config).
- Ensure existing tests continue to pass -- this is additive behavior.
