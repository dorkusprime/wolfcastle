# Write unknown field detection spec

Write a spec for unknown-field detection in config unmarshalling. The spec should define:

- Detection strategy: after the three-tier merge produces a map[string]any and before (or during) unmarshalling into the Config struct, compare the merged map's keys against the known JSON field names from the Config struct (and recursively for nested structs). Any key present in the map but absent from the struct's json tags is unknown.
- Approach options to evaluate:
  (a) json.Decoder with DisallowUnknownFields() -- simple but only works at the top-level unmarshal step, after merge. This is the Go stdlib approach.
  (b) Post-unmarshal diff: marshal the Config struct back to a map, then diff the keys of the input map vs the output map. Keys present in input but absent in output are unknown. More thorough for nested structures.
  Recommend (a) as the primary approach since it's idiomatic Go and the config struct already mirrors the JSON schema exactly. Note tradeoffs.
- Severity: warnings, not hard errors. Unknown fields should be reported to stderr (or included in a warnings list on the returned config) but should not prevent config loading. Users may have forward-compatible fields from a newer version.
- Warning format: 'config: unknown field "modles" in <tier>' -- include which tier file introduced the unknown field if possible, or at minimum report the field path in the merged config.
- Return type change: config.Load() may need to return (*Config, []Warning, error) or attach warnings to the Config struct, so callers can surface them. Evaluate which approach is less disruptive to existing callers.
- Per-tier detection: ideally detect unknown fields in each tier file individually (before merge) so the warning can name the source tier. This requires running DisallowUnknownFields or the diff approach against each tier's raw JSON independently.

Place the spec at docs/specs/ with the standard timestamped filename.
