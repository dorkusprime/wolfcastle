# Write config-reference.md

Create docs/humans/config-reference.md with every config field documented by section.

Source of truth: the Config struct in internal/config/types.go and the Defaults() function in internal/config/config.go. Read both thoroughly before writing.

Structure the page with a brief intro explaining this is the field-level reference, then one H2 per config section. Sections to cover (in this order): identity, models, pipeline, daemon, logs, retries, failure, git, overlap_advisory, summary, docs, validation, prompts, doctor, unblock, audit, archive, task_classes.

For each section, use an H3 per field. Each field entry must include:
- Type (e.g., string, int, bool, map[string]ModelDef)
- Default value (from Defaults())
- Description of what the field controls
- Example JSON snippet showing a non-default value

For nested structs (ModelDef, PipelineStage, PlanningConfig, etc.), document each nested field within the parent section.

Include a note at the top linking back to the Configuration Guide for context on how tiers and merge semantics work. Add cross-reference links where fields interact (e.g., pipeline.stages model references must exist in the models section).

Do NOT invent fields. Every field must come from the Config struct. If Defaults() doesn't set a value for a field, note the zero-value default.
