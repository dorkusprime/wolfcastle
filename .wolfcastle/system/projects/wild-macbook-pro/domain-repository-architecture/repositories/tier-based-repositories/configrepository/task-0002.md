# Implement ConfigRepository and tests

Implement ConfigRepository in internal/config/repository.go following the contract spec written in the previous task.

Key implementation details:
- The struct holds tiers (tierfs.Resolver) and root (string) fields
- NewConfigRepository(wolfcastleRoot) creates tierfs.New(filepath.Join(wolfcastleRoot, "system")) and delegates to NewConfigRepositoryWithTiers
- Load() reuses the existing merge logic: structToMap(Defaults()), then for each tier dir from tiers.TierDirs(), attempt to read config.json via os.ReadFile (not via tiers.Resolve, since we need all tiers not just highest-priority). DeepMerge each overlay. Marshal back to *Config. Run ValidateStructure
- WriteBase marshals the full *Config to indented JSON and calls tiers.WriteBase("config.json", data)
- WriteCustom and WriteLocal marshal map[string]any to indented JSON and write to tiers.TierDirs()[1]/config.json and tiers.TierDirs()[2]/config.json respectively, creating parent dirs as needed
- Keep the existing exported Load() function as a thin wrapper or deprecate it — check whether anything outside the config package calls it directly. If so, keep it as a convenience that constructs a temporary ConfigRepository

Tests (repository_test.go) should use testutil.NewEnvironment(t) to get a real tier filesystem, then exercise:
- Load() returns merged config with defaults
- Load() with custom tier overlay merges correctly
- Load() with local tier overlay takes highest priority
- WriteBase persists and is reflected in subsequent Load()
- WriteCustom writes partial overlay that merges on next Load()
- WriteLocal writes partial overlay that merges on next Load()
- Load() with malformed JSON in a tier returns error
- Load() with permission error propagates (if testable)

Do NOT modify the existing config.go Load() function or the configTiers variable yet. The repository wraps the merge logic independently. Cleanup of the old function happens in a later migration step.
