# Define Resolver interface and FS struct

Create internal/tierfs/tierfs.go with:

1. The Resolver interface with these methods (from spec):
   - Resolve(relPath string) ([]byte, error) — returns content from the highest-priority tier that has the file. Walk tiers from local -> custom -> base (highest to lowest priority). Return os.ErrNotExist (wrapped) if no tier has it.
   - ResolveAll(subdir string) (map[string][]byte, error) — collect all files in a subdirectory across tiers. Walk tiers lowest-to-highest so higher tier files overwrite lower. Only include .md files (matching existing ResolveAllFragments behavior). Key is the filename, value is content.
   - WriteBase(relPath string, data []byte) error — write to the base tier directory. Create parent directories as needed.
   - BasePath(subdir string) string — return the absolute path to a subdirectory within the base tier. Escape hatch for callers that need to walk the directory (e.g., embedded FS extraction).
   - TierDirs() []string — return absolute paths to all tier directories in resolution order (base, custom, local). Used by error messages that want to tell the user where they searched.

2. The FS struct:
   - Fields: root string (e.g., ".wolfcastle/system")
   - Constructor: New(root string) *FS
   - Tier names are hardcoded as ["base", "custom", "local"] — not configurable. This is the single source of truth for tier order (ADR-063).
   - FS implements Resolver.

Reference the existing tier resolution patterns in internal/pipeline/fragments.go (Tiers variable, ResolveFragment, ResolveAllFragments) and internal/config/config.go (configTiers). The tierfs package consolidates these into one place.

Error wrapping: use fmt.Errorf("tierfs: <operation>: %w", err) so callers can identify the source.
