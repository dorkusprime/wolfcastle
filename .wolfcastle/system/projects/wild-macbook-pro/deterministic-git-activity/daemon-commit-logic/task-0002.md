# Implement staging area preservation

Implement staging area preservation so the daemon's git add/commit cycle does not clobber the user's manually staged changes.

The spec suggests two approaches:
1. Stash-based: git stash --keep-index before commit, git stash pop after
2. Separate GIT_INDEX_FILE: use a temporary index file for daemon commits, leaving user's index untouched

Evaluate both approaches and choose the more reliable one. The separate GIT_INDEX_FILE approach is likely more robust because stash can conflict with existing stashes or fail on edge cases.

Implementation should wrap the commit function from task-0001: save user index state before staging, restore after commit. If the preservation mechanism fails, log a warning but do not block the commit (the commit itself is more important than preserving staged state).

Write tests that verify:
- User's staged changes survive a daemon commit
- User's unstaged changes are not affected
- The preservation mechanism handles clean working trees gracefully
