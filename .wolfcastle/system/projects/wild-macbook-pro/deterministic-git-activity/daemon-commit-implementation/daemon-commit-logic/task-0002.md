# Implement staging area preservation

The spec describes two approaches for preserving the user's staging area:

1. Stash approach: git stash --keep-index --quiet before staging, git stash pop --quiet after commit.
2. Separate index: use GIT_INDEX_FILE env var to write daemon commits to a temporary index, leaving the user's index untouched.

The separate index approach is more correct (stash can fail if the user has a merge conflict, and --keep-index has edge cases with new files). Evaluate both and implement whichever is more robust. If you choose the stash approach and find edge cases, switch to the separate index approach.

The implementation should:
- Save/restore user index state around every daemon commit
- Work correctly when there are no user-staged changes (no-op preservation)
- Work correctly when the working tree is clean (skip commit entirely)
- Not interfere with the user's ability to git add/commit manually between daemon iterations

If you choose the GIT_INDEX_FILE approach, the flow is:
1. Create a temp index: git read-tree HEAD into a temp file
2. Set GIT_INDEX_FILE=<temp> for all daemon git commands
3. git add in the temp index
4. git commit using the temp index
5. Remove temp index file

Document whichever approach you choose with an ADR via wolfcastle adr create.
