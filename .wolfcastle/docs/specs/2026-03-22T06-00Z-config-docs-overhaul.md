# Configuration Docs Overhaul

## Problem

The current configuration docs (`docs/humans/configuration.md`) explain the three-tier system conceptually but don't tell users how to actually customize Wolfcastle. Missing: where files live on disk, how to override specific fields, what the full config structure looks like, and practical examples of common tasks.

## Solution

Split the single config page into four pages:

### 1. Quickstart (`docs/humans/config-quickstart.md`)

Gets the user from "I just ran `wolfcastle init`" to a working customization in under a minute. Cover:

- **Change your model**: show the exact JSON to put in `system/custom/config.json` to switch from the default to a different model. Show the `wolfcastle config set` CLI alternative.
- **Add a custom stage**: show how to add a spec-review or custom stage to the pipeline.
- **Override a prompt**: show where to place a custom prompt file in `system/custom/prompts/` to override a base prompt.
- **Set a task class**: brief example with `--class coding/typescript`, link to the full Task Classes page for details.

Each section: 3-5 lines of prose, one JSON block or CLI command, done. No theory.

### 2. Configuration Guide (`docs/humans/configuration.md`, rewritten)

The "how it works" page. Cover:

- **Three-tier directory structure**: visual tree showing `system/base/`, `system/custom/`, `system/local/` with what each contains and what's gitignored vs. committed. Explain that base is regenerated by `init`, custom is committed (team-shared), local is gitignored (personal).
- **Merge semantics**: how tiers merge (higher overrides lower), with concrete examples. JSON objects deep-merge. Arrays replace entirely. `null` deletes a key. Show a before/after example of each.
- **Common customizations**: model configuration (defining models, switching the default), pipeline stages (dict format, stage_order, enabling/disabling), class overrides (adding a custom class, overriding a built-in class prompt), prompt overrides (placing a file in custom/ or local/ to replace a base prompt), rule fragments, audit scopes.
- **`wolfcastle config show`**: explain how to use it to inspect the merged result, with `--section` filtering. Show users they can dump the current config as a starting point for customization.
- **`wolfcastle config set/unset/append/remove`**: CLI alternatives to hand-editing JSON. Show each with examples.

### 3. Config Reference (`docs/humans/config-reference.md`)

Every config field, organized by section. For each field: type, default value, what it does, example. This is the "I need to know what `daemon.stall_timeout_seconds` does" page. Generated or maintained from the `Config` struct and `Defaults()` function.

Sections: `identity`, `daemon`, `pipeline`, `models`, `retries`, `failure`, `git`, `overlap`, `archive`, `task_classes`, `validation_commands`.

### 4. Task Classes (`docs/humans/task-classes.md`)

Dedicated page for the class system. Cover:

- **What classes are**: behavioral prompts that shape how the agent approaches work. Two layers: universal.md (always) + class-specific prompt. Classes don't restrict tools or change markers.
- **Using a class**: `wolfcastle task add --class coding/typescript` for languages, `--class coding/typescript/react` for frameworks. Show the fallback: `coding/typescript/react` tries the framework prompt first, falls back to `coding/typescript` if not found.
- **Supported languages**: full table of all built-in language classes with their class key and a one-line description. Organized by category (systems, JVM/.NET, web/scripting, specialized).
- **Supported frameworks**: full table of all built-in framework classes, grouped by parent language. Show the class key and what the prompt covers.
- **Non-language classes**: table of discipline classes (writing, research, architecture, audit, testing, design, devops, data, security) with descriptions.
- **Creating a custom class**: step-by-step guide. Add a `ClassDef` entry to `system/custom/config.json`, drop a `.md` file in `system/custom/prompts/classes/`, verify with `wolfcastle config show --section task_classes`. Show a complete example of creating a company-specific class (e.g., `coding/go-internal` with team conventions).
- **Overriding a built-in class**: place a file with the same path in `system/custom/prompts/classes/` to replace the base prompt. The user only needs to describe their coding conventions; no wolfcastle internals needed.
- **Creating a custom framework**: add a subdirectory under the parent language, drop the `.md` file, add the config entry. Show the fallback behavior.
- **The universal prompt**: explain what `universal.md` contains and that it applies to every task. Users can override it in `system/custom/` or `system/local/`.

## Cross-references

- Quickstart links to the Guide for explanation and Task Classes for class details.
- Guide links to the Reference for field details and Task Classes for class customization.
- Reference links back to the Guide for context on how fields interact.
- Task Classes links to the Guide for tier mechanics and the Reference for `task_classes` config fields.
- CLI command pages (`config-show.md`, `config-set.md`, `task-add.md`) link to relevant pages.
- README's Configuration section links to the Quickstart.

## Deep-linking pass

After the four pages are written, review every page in `docs/humans/` and the README for references to configuration that should now link to specific sections in the new pages. Examples:

- `how-it-works.md` mentions models, stages, and the pipeline. Link to the Guide's model and pipeline sections.
- `audits.md` mentions audit scopes and configuration. Link to the Reference's audit-related fields.
- `failure-and-recovery.md` mentions configurable thresholds. Link to the Reference's `failure` and `retries` sections.
- `collaboration.md` mentions git config and worktree settings. Link to the Reference's `git` section.
- CLI pages (`config-show.md`, `config-set.md`, etc.) should link to the Guide for context on what they're modifying.
- README's Configuration section should link to the Quickstart as the entry point.

Any mention of "three tiers," "base/custom/local," "config.json," or specific config field names in any human doc should link to the appropriate new page and section.

## What This Does Not Cover

- Prompt authoring internals (how ContextBuilder assembles prompts, covered by the task classes spec).
- State file internals (covered by the state machine spec).
- CLI command reference (covered by `docs/humans/cli/`).
