# ADR-063: Three-Tier Configuration

## Status
Accepted

## Date
2026-03-14

## Context

Wolfcastle's prompt and rule files already follow a three-tier resolution pattern: `base/` (regenerated defaults, gitignored), `custom/` (team overrides, committed), and `local/` (personal preferences, gitignored). Configuration, however, remained a special case: a root `config.json` (committed) and `config.local.json` (gitignored) living directly in the `.wolfcastle/` directory.

This inconsistency meant that new contributors had to learn two different mental models for override behavior. It also meant that `config.json` served double duty as both the default template and the team override surface, creating friction during `init --force`: the re-scaffold logic needed a DeepMerge hack to update stale defaults without clobbering user customizations in the same file.

## Decision

Align configuration with the three-tier pattern used by prompts and rules. The config resolution chain becomes:

```
hardcoded defaults <- base/config.json <- custom/config.json <- local/config.json
```

**base/config.json** is written by `Scaffold` and regenerated by `ReScaffold`. It contains the full compiled defaults (minus identity) and is gitignored. Because it is always overwritten, new config keys introduced in later versions appear automatically after `init --force`.

**custom/config.json** is the team override surface. It is committed to the repository. `Scaffold` writes an empty `{}` here; teams add their model definitions, pipeline stages, validation commands, and other shared settings.

**local/config.json** holds personal overrides and engineer identity. It is gitignored. `Scaffold` writes the detected identity here; `ReScaffold` refreshes identity while preserving any other keys the engineer has added.

`ReScaffold` includes a migration path: if old-style root `config.json` exists, it is moved to `custom/config.json`; if `config.local.json` exists, its contents are merged into `local/config.json`. Both old files are removed after migration.

## Consequences

- Configuration follows the same three-tier pattern as prompts and rules. One mental model for all override behavior.
- The DeepMerge hack in `ReScaffold` (overlay new defaults onto existing `config.json` without clobbering customizations) is eliminated. `base/config.json` is simply overwritten with fresh defaults every time.
- Existing installs are migrated automatically on `init --force`. The migration moves `config.json` to `custom/config.json` and merges `config.local.json` into `local/config.json`.
- The `.gitignore` template no longer needs the `!config.json` exception; `base/` and `local/` are already gitignored, and `custom/` is already committed.
- Teams gain a clean separation: `custom/config.json` is the single file reviewed in PRs for team config changes, free of regenerated defaults that would otherwise create noisy diffs.

**Amendment (ADR-077):** The three tiers moved under `.wolfcastle/system/` (`system/base/`, `system/custom/`, `system/local/`). The merge semantics and override order are unchanged.

**Amendment:** `internal/tierfs` is the single source of truth for tier names and resolution order. All packages that need tier-aware paths (`config`, `pipeline`, `project`, `cmd/audit`) derive their tier references from `tierfs.TierNames` and `tierfs.SystemTierPaths()` rather than maintaining independent lists. The `pipeline` package's legacy free functions (`ResolveFragment`, `ResolveAllFragments`, `ResolvePromptTemplate`) now delegate to a `tierfs.Resolver` internally.
