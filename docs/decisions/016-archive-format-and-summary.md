# ADR-016: Archive Format with Deterministic Rollup and Model Summary

## Status
Accepted

## Date
2026-03-12

## Context
Completed work needs to graduate from live state (`projects/`) to a permanent, committed, merge-conflict-proof record (`archive/`). Ralph captured learnings in STATUS.md and audit results in audit.md, but these were scattered across the tree and only readable by navigating the project structure.

## Decision

### Breadcrumbs During Execution
As tasks execute, the model writes breadcrumbs via script calls. These should be richer and more explanatory than Ralph's terse notes, because they serve as the raw material for the archive entry.

### Node Completion Flow
When a node completes:

1. **Tasks execute** — model writes breadcrumbs along the way
2. **Audit task runs** — verifies work, fixes gaps, escalates
3. **Summary stage (opt-out)** — a configurable (typically lower-tier) model writes a brief plain-language summary of what the node accomplished and why it matters. This runs as its own pipeline stage after audit, so the model has the full picture. Enabled by default but can be disabled in config for users who don't want the token overhead.
4. **Archive rollup** — `wolfcastle archive add --node <path>` deterministically generates the archive entry. If summary is enabled: model's summary at the top, then breadcrumbs, audit results, and completion metadata. If summary is disabled: breadcrumbs, audit results, and metadata only — still a useful record.

### Archive Entry Structure
Each archive entry is a Markdown file with a timestamp filename (per ADR-011):

```
.wolfcastle/archive/2026-03-12T18-45Z-fire-implementation-complete.md
```

Contents (generated by script, not by model):
- **Summary** — model-written plain-language summary (from step 3)
- **Breadcrumbs** — chronological record of task-level work
- **Audit results** — scope verified, gaps found and fixed, escalations
- **Metadata** — node path, completion timestamp, engineer identity, branch

### No Extra Model Call for Rollup
The archive script generates everything deterministically from JSON state except the summary, which was already written by the model in step 3. No separate summarization model call at archive time.

## Consequences
- Archive entries are self-contained, human-readable records of completed work
- The summary gives context that breadcrumbs alone don't provide
- Summary is opt-out — disabled via config for cost-sensitive users, archive still works without it
- When enabled, summary model tier is configurable — cheaper models work fine for this
- Merge-conflict-proof by design (unique timestamped filenames per ADR-011)
- Archive generation is deterministic and testable (script, not model, assembles the entry)
- Breadcrumb quality matters — the model's per-task notes directly feed the permanent record
