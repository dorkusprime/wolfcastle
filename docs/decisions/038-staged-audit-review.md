# ADR-038: Staged Audit Review Workflow

**Status:** Accepted

**Date:** 2026-03-14

## Context

ADR-029 introduced `wolfcastle audit run`, a model-driven codebase audit that
discovers scopes, invokes a model, and presents findings for immediate
interactive approval. The original implementation offered three inline choices:
approve all, skip, or manual. This created several problems:

1. **Ephemeral findings.** If the user chose "skip," findings existed only in
   terminal scrollback. A ctrl-C during review lost everything.
2. **No async review.** The synchronous prompt required immediate decisions.
   A teammate could not review findings generated by someone else.
3. **No audit trail.** There was no durable record of what was approved,
   rejected, or why: undermining the auditability that the rest of the
   system carefully preserves.
4. **All-or-nothing approval.** "Approve all" created projects for every
   finding without individual consideration; "manual" left the user to
   reconstruct findings from memory.

## Decision

Replace the inline interactive approval with a staged review pipeline backed
by two durable JSON files:

- **`audit-state.json`** (originally named `audit-review.json`): the current pending batch of findings. Created by
  `audit run`, consumed by `audit pending`, `audit approve`, and `audit reject`.
  Removed once all findings are decided.
- **`audit-review-history.json`**: append-only log of completed batches with
  every approve/reject decision, timestamps, scopes, and created node addresses.
  Subject to retention (default: 100 entries, 90 days).

The workflow becomes:

1. `wolfcastle audit run [--scope ...]`: invokes the model, parses findings
   into a `Batch` with `Finding` entries (each initially `pending`), saves to
   `audit-review.json`. Refuses to overwrite an existing pending batch.
2. `wolfcastle audit pending`: lists undecided findings with IDs, titles,
   and description previews.
3. `wolfcastle audit approve <id | --all>`: marks finding(s) as approved,
   creates leaf projects in the work tree with descriptions extracted from
   the finding.
4. `wolfcastle audit reject <id | --all>`: marks finding(s) as rejected
   without creating anything.
5. `wolfcastle audit history`: displays past batches with decision summaries.

When the last finding in a batch is decided, the batch is automatically
archived to history and the pending file is removed.

Findings are parsed from model output using the same heading/bold-item
heuristics as before, but now each finding carries a description (the text
between headings) in addition to its title, providing richer context for
the reviewer.

The review types (`Batch`, `Finding`, `HistoryEntry`, `Decision`, `History`)
and I/O (`LoadBatch`, `SaveBatch`, `LoadHistory`, `SaveHistory`,
`EnforceRetention`) live in `internal/state/` (moved from the original
`internal/review/` per ADR-058).

## Consequences

- **Durability.** Findings survive across sessions, crashes, and ctrl-C.
  The pending file is the source of truth until all findings are decided.
- **Async review.** Any engineer with access to `.wolfcastle/` can review
  findings generated by another. The review commands are stateless: they
  read and write the same JSON files.
- **Audit compliance.** Every decision is recorded with timestamp, finding
  title, action, and created node. The history file provides a complete
  audit trail of all codebase audit decisions.
- **No accumulation.** History is subject to retention limits (max entries
  and max age). The pending batch file is removed after completion, so only
  one batch exists at a time.
- **Breaking change.** The inline `[a/s/m]` approval flow is removed. Users
  who relied on the synchronous approve-all workflow must now use
  `audit approve --all` as a second step.
