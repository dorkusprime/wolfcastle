# Wolfcastle TUI

> **SUPERSEDED by `2026-04-06T01-00Z-tui-v01.md`.** This spec is historical. Do not use it for implementation decisions.

A terminal user interface for observing and commanding the Wolfcastle daemon, built on Bubbletea.

The `wolfcastle follow` and `wolfcastle status` commands work, but they are separate views into the same system. The TUI combines them into a single, persistent interface: a live dashboard where you can watch the daemon hunt, browse the project tree, read logs, manage the inbox, intervene on blocked tasks, and tune configuration without leaving the terminal.

## Governing ADRs

- ADR-020: Daemon Lifecycle and Process Management
- ADR-021: CLI Command Surface
- ADR-024: Distributed State Files, Task Working Documents, and Runtime Aggregation
- ADR-025: Wolfcastle Doctor (structural validation and repair)
- ADR-063: Three-Tier Configuration

## 1. Purpose and Goals

The TUI answers three questions at a glance:

- **What is the daemon doing right now?** (Dashboard, log stream)
- **What does the project tree look like?** (Tree browser, node/task detail)
- **What needs my attention?** (Blocked tasks, open gaps, escalations, inbox)

From v0.1 it is an observation tool. From v0.2 onward it becomes a control surface, growing through inbox writes, task management, interactive debugging, and configuration editing across successive versions.

## 2. Layout and Panes

The interface uses a split-pane layout with a header bar, two main content areas, and a footer.

```
┌─────────────────────────────────────────────────────────────────┐
│ WOLFCASTLE v0.1.0                      Daemon: hunting (PID X)  │
│ 12 nodes: 4● 3◐ 3◯ 2☢"  Audit: 2 passed, 1 gap open             │
├──────────────────────┬──────────────────────────────────────────┤
│                      │                                          │
│   Project Tree       │   Detail Pane                            │
│                      │                                          │
│ ▸ auth-system    ●   │   (varies by context: node detail,       │
│   ▸ login     ◐      │    task detail, log stream, inbox,       │
│   ▸ oauth     ◯      │    audit trail, prompt inspector,        │
│ ▸ api-gateway ☢  ●   │    config, unblock session, ...)         │
│   ▸ rate-limit ☢     │                                          │
│                      │                                          │
├──────────────────────┴──────────────────────────────────────────┤
│ [q]uit  [d]ashboard  [l]ogs  [i]nbox  [/]search  [?]help        │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 Header Bar (v0.1)

Two lines, always visible.

**Line 1:** Product name, version, daemon status. The daemon status reads the PID file and probes the process, same as `getDaemonStatus` does today.

**Line 2:** Aggregate counts. Node status breakdown using the four-glyph vocabulary (see 2.9). Audit summary: counts by AuditStatus, plus open gap and escalation totals.

**Narrow-width truncation:** When the terminal is too narrow to fit both header lines at full width, the header truncates from the right. Line 1 keeps the product name and daemon status, dropping the PID. Line 2 keeps the node count total, dropping per-status breakdowns. Below 40 columns, line 2 is hidden entirely. The header never wraps.

### 2.2 Left Pane: Project Tree (v0.1)

A navigable tree built from the `RootIndex`. Each entry shows the node name and a status glyph. Orchestrator nodes are collapsible; expanding one reveals its children. Leaf nodes show their tasks when expanded.

The tree respects the index's `Root` ordering for top-level nodes and `Children` ordering within orchestrators. Each entry also carries the `IndexEntry.DecompositionDepth` from the index; nodes at depth > 1 render with an indentation marker to convey nesting depth at a glance.

**Current target indicator (v0.1):** The node and task the daemon is currently executing get a `▶` prefix in the tree, rendered in bright yellow. This is derived from the most recent `stage_start` log entry that names a node/task pair. When the daemon is idle (no active stage), no indicator is shown.

**Tree filtering and sorting (v0.2):** `F` cycles through tree sort/filter modes: default order (index ordering), blocked-first (all blocked nodes float to top), gaps-first (nodes with open audit gaps float to top). The active sort mode displays in the tree pane header. Filtering does not hide nodes; it reorders them.

Width: fixed at roughly 30% of the terminal, with a minimum of 24 columns. If the terminal is too narrow (under 60 columns), the tree pane collapses and the detail pane fills the screen; pressing `t` toggles the tree back.

### 2.3 Right Pane: Detail (v0.1+)

The detail pane's content depends on what's selected and which mode is active. Modes are introduced progressively across versions:

| Mode                 | Trigger                    | Version | Content                                                                                |
| -------------------- | -------------------------- | ------- | -------------------------------------------------------------------------------------- |
| Dashboard            | `d` key or startup default | v0.1    | Iteration progress, daemon health, recent activity summary                             |
| Node detail          | Enter on an orchestrator   | v0.1    | Node name, type, status, decomposition depth, specs, children summary, audit state (including scope criteria, started/completed timestamps), breadcrumbs, gaps (with source, fixed-by, fixed-at), escalations (with source gap ID) |
| Task detail          | Enter on a task            | v0.1    | Full task view (see 2.10)                                                              |
| Log stream           | `l` key                    | v0.1    | Live NDJSON log output, formatted and colorized                                        |
| Inbox                | `i` key                    | v0.2    | Inbox items list with status, text input for adding new items                          |
| Audit trail          | `A` key on a node          | v0.2    | Full audit state for the selected node (see 2.11)                                      |
| Task management      | Context menu on a task     | v0.3    | Block/unblock, add task, trigger decomposition                                         |
| Unblock session      | `u` key on a blocked task  | v0.4    | Embedded unblock pane (see 2.12)                                                       |
| Prompt inspector     | `P` key on a task          | v0.5    | Assembled prompt preview (see 2.13)                                                    |
| Config viewer/editor | `C` key                    | v0.5    | Three-tier config display (see 2.14)                                                   |
| Archive browser      | `B` key                    | v0.5    | Completed project archives (see 2.15)                                                  |

The detail pane is a `bubbles/viewport` (scrollable). In log stream mode it supports both follow mode (auto-scroll to bottom) and manual scrolling.

### 2.4 Footer (v0.1)

A single line showing available key bindings for the current context. The bindings change depending on which pane has focus and which mode the detail pane is in.

### 2.5 Notification Area (v0.2)

Toast notifications appear in the upper-right corner of the detail pane, overlaying content briefly before fading. Toasts announce state transitions:

- Task complete: `Target eliminated: {addr}/{task}`
- Task blocked: `Blocked: {addr}/{task}`
- Intake filed: `New intel received.`
- Audit gap opened: `Gap opened: {node} {gap_type}`
- New node created: `New target acquired: {addr}`
- Error: `State file unreadable. Retrying.`

Toasts auto-dismiss after 3 seconds. A toast queue holds up to 5 pending notifications; older ones are dropped if the queue overflows.

**Sound/bell (v0.3):** Configurable terminal bell on errors or task completion. Off by default. Enabled via `tui.bell` in `local/config.json`.

**Desktop notifications (v0.5):** Optional integration with OS notification center (macOS `osascript`, Linux `notify-send`). Off by default. Enabled via `tui.desktop_notifications` in `local/config.json`.

### 2.6 Initialization and Cold Start (v0.1)

The TUI must handle launching into a system with no prior state: no daemon running, no PID file, no root index, no log files. This is the cold start scenario, and it is the first impression for new users.

**Cold start behavior:**

1. Header line 1 shows `Daemon: standing down`. Header line 2 shows `0 nodes`.
2. The tree pane displays `No targets. Feed the inbox.` (same empty state text as section 11.2).
3. The detail pane opens to the dashboard, which reads: `No transmissions. The daemon has not spoken.`
4. The footer shows available key bindings as normal. The TUI is fully navigable even with no data.

**Loading and spinner states:** When the TUI is reading state files on startup or after a file-watch event triggers a reload, a spinner glyph (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`, cycling) appears in the header next to the daemon status. The spinner displays for any async read that takes longer than 100ms. It disappears once the read completes and the view re-renders.

**Degraded state:** If a state file exists but cannot be parsed (corrupt JSON, unexpected schema version), the TUI does not crash. It shows the last known good state for that file and displays a persistent error bar above the footer: `State corruption detected: {filename}. Run wolfcastle doctor.` This bar remains visible until the file is successfully re-read or the user dismisses it with `Esc`. See also section 11.7 for error display voice.

### 2.7 Command Palette (v0.2)

`Ctrl+P` or `:` opens a command palette overlay, centered in the detail pane.

The palette provides fuzzy search over:

- All TUI commands (switch mode, toggle pane, change filter)
- Node addresses (jump to any node in the tree)
- Task IDs (jump to any task)
- Log entries by trace ID

Typing narrows the result list. `Enter` executes the selected command or navigates to the selected item. `Esc` dismisses the palette.

The palette uses `bubbles/textinput` for the search field and a filtered list below it. Fuzzy matching follows the same algorithm as VS Code's command palette: substring match with character-skip scoring, case-insensitive.

### 2.8 Search (v0.1+)

**Pane-local search (v0.1):** `/` opens an incremental search bar at the bottom of the current pane. Typing highlights matches in the viewport. `n` and `N` move between matches. `Esc` dismisses the search bar. When no matches are found, the search bar reads: `No matches. Adjust your aim.`

**Global search (v0.3):** `Ctrl+/` opens a global search that searches across logs, task descriptions, audit trails, and node names. Results appear in the detail pane as a navigable list grouped by source. Enter on a result navigates to the source item. When global search finds nothing, the result pane reads: `No matches. Adjust your aim.` Some terminals do not transmit `Ctrl+/`. Fallbacks: `Ctrl+_` (which most terminals encode identically) and the command palette command `:search`.

**Regex support (v0.3):** Search patterns are treated as literals by default. Prefix with `r/` to enable regex (e.g., `r/task-\d{3}`).

### 2.10 Task Detail View (v0.1, expanded v0.3)

The task detail view shows the full picture for a single task:

**v0.1 (read-only):**

- Task ID, title, description
- Status and status glyph
- Failure count and block reason (if any)
- `is_audit` and `needs_decomposition` flags
- Task breadcrumbs: the `Task.Breadcrumbs` field is a `[]string` (plain text tags). These render as a comma-separated inline list. Distinct from audit breadcrumbs (see section 2.11), which are `[]Breadcrumb` structs with timestamp, task, and text fields.

**v0.3 additions (task management):**

- State history: a timeline of status transitions with timestamps
- Full `.md` content from the task definition
- Action bindings: `b` to block, `u` to unblock, `D` to trigger decomposition

### 2.11 Audit Trail View (v0.2)

Triggered by pressing `A` on a selected node. Shows the full audit state:

- **Status and timestamps:** `AuditState.Status`, `AuditState.StartedAt`, `AuditState.CompletedAt`. Timestamps render as relative time (e.g., "3m ago") with the absolute time on hover or in a tooltip line below.
- **Scope:** The audit scope definition for this node, including `AuditScope.Description`, `Files`, `Systems`, and `Criteria`. Each criteria item renders as a checklist row.
- **Breadcrumbs:** All `AuditState.Breadcrumbs` (these are `[]Breadcrumb` structs, each with `Timestamp`, `Task`, and `Text`), ordered chronologically, with source task IDs. Distinct from the plain-string `Task.Breadcrumbs` shown in the task detail view.
- **Gaps:** Open and closed gaps, with status indicators (`⊘` open, `⏸` closed). Each gap shows `Gap.ID`, `Description`, `Source` (the audit task that found it), `Status`, and, if fixed, `FixedBy` and `FixedAt`.
- **Escalations:** Active escalations with their reasons, timestamps, `Escalation.SourceNode`, `Escalation.SourceGapID` (linking back to the originating gap), and resolution status.
- **Summary:** The generated `AuditState.ResultSummary` if one exists

Gap status indicators use color: red for open gaps, green for closed. The view is scrollable and follows the same viewport pattern as other detail views.

### 2.12 Unblock Session Pane (v0.4)

Pressing `u` on a blocked task opens an embedded unblock session in the detail pane. This is the same workflow as `wolfcastle unblock` but rendered inside the TUI.

The pane contains:

- **Context header:** Task ID, block reason, failure count, last error output
- **Chat area:** A scrollable viewport showing the conversation with the model
- **Input line:** A readline-style text input at the bottom, with history (up/down arrows recall previous inputs)

The unblock session calls the same `UnblockWorkflow` backend that the CLI command uses. Messages stream into the chat area as they arrive. The user can type follow-up questions, paste error output, or issue commands like `/retry` to re-attempt the task.

`Esc` exits the unblock session and returns to the task detail view. The session state (conversation history) is preserved for the duration of the TUI process; returning to the same task's unblock session resumes where you left off.

### 2.13 Prompt Inspector (v0.5)

Pressing `P` on a task shows the assembled prompt that would be sent to the model. The view contains:

- **Stage breakdown:** Each stage of the prompt assembly shown as a collapsible section (system prompt, context, task description, constraints, etc.)
- **Token count estimate:** Approximate token count per stage and total
- **Diff from default:** Highlights where custom or local config overrides change the default prompt

The prompt inspector is read-only. It calls the same prompt assembly code that the daemon uses but renders the result instead of sending it.

### 2.14 Config Viewer/Editor (v0.5)

Pressing `C` opens the configuration view. Three columns show base, custom, and local config tiers side by side, with the merged result highlighted.

**Viewer features:**

- Fields that differ from the base tier are highlighted in yellow (custom) or cyan (local)
- `null` overrides are shown explicitly with a strikethrough on the base value
- Scroll and search within the config view

**Editor features (v0.5):**

- Press `e` on a field to edit its value in the local tier
- Validation runs on edit: type checking, range validation, enum validation
- Changes write to `local/config.json` only. Base and custom tiers are read-only.
- A confirmation prompt appears before writing: `Overwrite local config? (y/n)`

### 2.15 Archive Browser (v0.5)

Pressing `B` opens a list of completed project archives. Each entry shows the project name, completion date, and node count. Selecting an archive loads its tree in read-only mode using the same tree and detail views.

### 2.16 ADR/Spec Browser (v0.5)

Pressing `S` opens a file browser rooted at the `docs/` directory. Markdown files render with basic formatting (headers, bold, lists, code blocks) using lipgloss styling. Navigation is the same as the project tree: `j`/`k` to move, `Enter` to open, `Esc` to go back.

### 2.17 Multi-Engineer View (v0.6)

Pressing `M` switches to a multi-engineer layout. The tree pane splits into columns, one per engineer namespace (same data as `status --all`).

Each column shows that engineer's tree with status glyphs. An overlap advisory panel at the bottom flags nodes that multiple engineers are working on simultaneously, with a color-coded severity indicator:

| Overlap                     | Indicator | Meaning                    |
| --------------------------- | --------- | -------------------------- |
| Same node, different tasks  | Yellow    | Low risk, monitor          |
| Same task, both in_progress | Red       | Conflict likely, intervene |
| Same node, one blocked      | Dim       | Informational only         |

The multi-engineer view is read-only in all versions. Write operations apply to the active engineer namespace only.

### 2.18 Worktree Management (v0.7)

Pressing `W` opens the worktree view. The pane shows:

- **Active worktrees:** List of git worktrees with their branch names, paths, and status (clean, dirty, ahead/behind)
- **Branch graph:** A simplified branch relationship visualization showing where worktree branches diverge from and merge into main
- **Actions:** `n` to create a new worktree (prompts for branch name), `s` to switch the daemon to a different worktree, `x` to delete a worktree (with confirmation)

Worktree operations shell out to `git worktree` commands. The daemon switch operation updates the active namespace and restarts the watcher subscriptions.

## 4. Data Sources

Each pane reads from specific files and APIs. All reads go through the existing `Store` or direct file I/O. Write operations are introduced in v0.2 and expand through later versions.

### 4.1 Header Bar (v0.1)

- **Daemon status:** Read `$WOLFCASTLE_DIR/wolfcastle.pid`, parse PID, probe with `signal(0)`. Same logic as `getDaemonStatus`.
- **Aggregate counts:** Derived from the `RootIndex` (node status counts) and a pass over leaf `NodeState` files (audit counts, open gaps, open escalations).

### 4.2 Project Tree (v0.1)

- **Tree structure:** `Store.ReadIndex()` returns the `RootIndex`. The `Nodes` map provides every node's name, type, status, parent, and children. The `Root` slice gives top-level ordering.
- **Task list (leaf expansion):** `Store.ReadNode(addr)` returns the `NodeState` with its `Tasks` slice.

### 4.3 Node / Task Detail (v0.1)

- **Node detail:** `Store.ReadNode(addr)` for the full `NodeState`, including `Audit` (scope, breadcrumbs, gaps, escalations), `Children`, `Tasks`, `DecompositionDepth`, and `Specs` (linked spec file paths, rendered as clickable references in the detail view).
- **Task detail:** Comes from the parent `NodeState.Tasks` slice, filtered by task ID.

### 4.4 Log Stream (v0.1)

- **Log files:** Read from `$WOLFCASTLE_DIR/logs/*.jsonl`. Use `logging.LatestLogFile` to find the current file, then tail it.
- **Parsing:** Each line is a JSON object. The TUI parses the same fields as `formatAndPrintLogLine`: `type`, `level`, `trace`, `stage`, `message`, `text`, `node`, `task`, `exit_code`, `error`, `count`, `reason`.
- **Log `type` rendering:** The following `type` values are recognized, each with its own formatting:

| `type` value        | Rendering                                                                 |
| ------------------- | ------------------------------------------------------------------------- |
| `stage_start`       | `[{stage}] Starting {node}/{task}` (or `[{stage}] Starting` if no node)  |
| `stage_complete`    | `[{stage}] Complete (exit={exit_code})`                                   |
| `stage_error`       | `[{stage}] Error: {error}` in red                                        |
| `assistant`         | Raw `text` content, printed verbatim                                      |
| `failure_increment` | `[failure] Task {task} failure count: {count}`                            |
| `auto_block`        | `[blocked] Task {task} auto-blocked: {reason}` in red                     |
| `daemon_start`      | `Daemon started` in bold                                                  |
| (unrecognized)      | `[{type}] {message}` in dim, falling back to the raw JSON if no message  |

Previously, unrecognized types were silently dropped. The TUI renders them with the generic fallback so that new log types added to the daemon are visible without a TUI update.

- **Formatting:** Colorized by level (debug=dim, info=white, warn=yellow, error=red). Trace IDs shown as prefixes: `[exec-0042]`, `[intake-10003]`.
- **Historical lines:** On first load, show the last 50 lines from the current log file. Then stream new lines as they appear.

### 4.5 Inbox (v0.2)

- **Read:** `Store.ReadInbox()` returns `InboxFile` with its `Items` slice.
- **Write (add item):** `Store.MutateInbox()` appends a new `InboxItem` with timestamp, text, and status `"new"`.

### 4.6 Task Mutations (v0.3)

- **Block task:** `Store.MutateNode(addr)` sets the target task's status to `blocked` and records the block reason.
- **Unblock task:** `Store.MutateNode(addr)` clears the block reason and sets the task status back to `not_started`.
- **Add task:** `Store.MutateNode(addr)` appends a new task to the node's `Tasks` slice.
- **Trigger decomposition:** `Store.MutateNode(addr)` sets `needs_decomposition: true` on the target task. The daemon picks this up on its next iteration.

### 4.7 Unblock Session (v0.4)

- **Backend:** Calls the same `UnblockWorkflow` used by `wolfcastle unblock`. The TUI provides a `tea.Cmd` wrapper that runs the workflow in a goroutine and streams messages back via `tea.Msg`.
- **Session state:** Conversation history lives in memory for the TUI's lifetime. Not persisted to disk.

### 4.8 Prompt Inspector (v0.5)

- **Assembly:** Calls the prompt assembly pipeline with the target task's context but does not execute the model call. Returns the assembled prompt stages as structured data.
- **Token estimation:** Uses a simple whitespace tokenizer for approximate counts (not model-specific tokenization; close enough for debugging).

### 4.9 Config (v0.5)

- **Read:** Loads all three config tiers (base, custom, local) independently, plus the merged result.
- **Write:** Edits write to `local/config.json` only, using the same atomic-write pattern as `Store`.

### 4.10 Multi-Engineer (v0.6)

- **Namespace enumeration:** Reads all subdirectories of `$WOLFCASTLE_DIR/projects/` as engineer namespaces.
- **Per-namespace state:** Calls `Store.ReadIndex()` for each namespace. Each namespace gets its own `Store` instance.
- **Overlap detection:** Compares in-progress tasks across namespaces by node address. A task appearing as `in_progress` in two namespaces is flagged.

### 4.11 Worktree (v0.7)

- **List:** Shells out to `git worktree list --porcelain` and parses the output.
- **Create/delete:** Shells out to `git worktree add` / `git worktree remove`.
- **Branch status:** `git log --oneline main..HEAD` and `git log --oneline HEAD..main` for ahead/behind counts.

## 5. Real-Time Update Strategy

The TUI needs to reflect changes as the daemon works. Two mechanisms, chosen per data type:

### 5.1 File Watching with fsnotify (v0.1)

Primary mechanism. Watch these paths:

| Path                                       | Triggers update of                    |
| ------------------------------------------ | ------------------------------------- |
| `$WOLFCASTLE_DIR/wolfcastle.pid`           | Header daemon status                  |
| `$PROJECTS_DIR/state.json` (root index)    | Tree structure, header counts         |
| `$PROJECTS_DIR/*/state.json` (node states) | Node detail, task detail, audit info  |
| `$PROJECTS_DIR/inbox.json`                 | Inbox pane                            |
| `$WOLFCASTLE_DIR/logs/*.jsonl`             | Log stream (new data in current file) |

When fsnotify fires a write event, the TUI sends a `tea.Msg` into the Bubbletea event loop. The relevant sub-model re-reads its data and the view re-renders.

For the log directory, also watch for new file creation events. When a new `.jsonl` file appears (new iteration), switch the log stream to tail it.

**Multi-engineer watching (v0.6):** When the multi-engineer view is active, the watcher adds `state.json` paths for all engineer namespaces. Events from non-active namespaces update only the multi-engineer view, not the primary tree.

### 5.2 Polling Fallback (v0.1)

fsnotify does not work on all filesystems (notably some network mounts). The TUI also runs a low-frequency poll (every 2 seconds) that re-checks the root index and PID file. If fsnotify is delivering events, the poll is a no-op (compare file mod times before re-reading).

### 5.3 Log Tailing (v0.1)

The log stream updates at a higher frequency than tree state and gets special treatment.

The TUI maintains a file offset (same approach as `tailFileStreaming` in `follow.go`). On each fsnotify event or poll tick, it seeks to the offset, reads new lines, parses and appends them to the viewport buffer, and updates the offset.

When the daemon starts a new iteration (detected by a new file appearing in the logs directory), the TUI prints a separator line and switches to the new file.

Buffer size: keep the last 10,000 parsed log lines in memory. Older lines are discarded. This prevents unbounded memory growth during long daemon runs.

### 5.4 Update Debouncing (v0.1)

State file changes can come in bursts (the daemon writes the node state, then the index, then propagates to parents, all within milliseconds). The TUI debounces fsnotify events with a 100ms window: collect all events in the window, then re-read only the affected files once.

### 5.5 Notification Triggering (v0.2)

State diffs drive the notification system. When a `StateUpdatedMsg` or `NodeUpdatedMsg` arrives, the TUI compares the new state against the previous snapshot:

- Task status changed to `complete`: fire a completion toast
- Task status changed to `blocked`: fire a blocked toast
- Inbox item count increased: fire an inbox toast
- Audit gap opened: fire a gap toast
- New node appeared in `RootIndex.Nodes` (inbox processing created it): fire a new-node toast

The diff is lightweight: compare status fields and counts, not deep equality on the full state.

### 6.2 Message Types

Custom `tea.Msg` types for data updates:

**v0.1:**

- `StateUpdatedMsg` carries a fresh `RootIndex` and triggers tree/header re-render.
- `NodeUpdatedMsg` carries a fresh `NodeState` for a specific address.
- `LogLinesMsg` carries a batch of new parsed log lines.
- `DaemonStatusMsg` carries the daemon status string.
- `NewLogFileMsg` signals iteration rollover with the new file path.

**v0.2:**

- `InboxUpdatedMsg` carries a fresh `InboxFile`.
- `ToastMsg` carries a notification to display.

**v0.3:**

- `TaskMutatedMsg` confirms a task mutation completed successfully.
- `DecompositionTriggeredMsg` confirms decomposition was flagged.

**v0.4:**

- `UnblockStreamMsg` carries a chunk of model output for the unblock session.
- `UnblockCompleteMsg` signals the unblock session finished.

**v0.5:**

- `ConfigLoadedMsg` carries all three config tiers and the merged result.
- `ConfigWrittenMsg` confirms a local config edit was persisted.
- `PromptAssembledMsg` carries the assembled prompt stages.

**v0.6:**

- `MultiEngineerUpdatedMsg` carries state for all engineer namespaces.
- `OverlapDetectedMsg` carries overlap advisory data.

**v0.7:**

- `WorktreeListMsg` carries the parsed worktree list.
- `WorktreeActionMsg` confirms a worktree create/delete/switch completed.

### 6.3 Subscriptions

A single `tea.Sub` manages the fsnotify watcher and polling loop. It sends the appropriate message types into the Bubbletea event loop. The subscription is started once in `Init()` and runs for the lifetime of the TUI.

The subscription goroutine:

1. Sets up fsnotify watches on the relevant paths.
2. Enters a select loop on fsnotify events, poll timer, and a done channel.
3. On each event, reads the affected file(s) and sends a typed message via the program's `Send` method.

**v0.6 extension:** When multi-engineer mode activates, the subscription adds watches for all engineer namespaces. When it deactivates, it removes the extra watches.

### 6.4 Focus Management

The `TUIModel` tracks which pane has focus (`tree` or `detail`). Key events route to the focused pane's `Update` method first. Global keys are handled at the top level before delegation.

The `Tab` key cycles focus. Visual indicator: the focused pane gets a highlighted border (lipgloss border style swap).

Overlay models (help, command palette, search) capture focus entirely when active. They handle all input until dismissed.

### 6.5 Responsive Sizing

`tea.WindowSizeMsg` triggers re-layout. The `TUIModel` recalculates column widths (tree: 30%, detail: 70%, respecting minimums) and propagates new dimensions to sub-models. Each sub-model's `View()` respects its allocated width and height.

Below 60 columns total width, the tree pane hides and the detail pane goes full-width.

**Multi-engineer layout (v0.6):** The tree pane splits into equal-width columns, one per namespace, with a minimum of 20 columns each. If the terminal cannot fit all namespaces, horizontal scrolling is enabled: `H` and `L` (Shift+h, Shift+l) scroll the engineer columns left and right by one column width. A position indicator in the multi-engineer header shows which columns are visible (e.g., `[2-4 of 6]`).

## 7. Performance

### 7.1 Memory Budget

The TUI process targets a 50MB memory ceiling under normal operation (trees under 100 nodes, single active log file). The major consumers:

| Component                     | Budget | Notes                                                       |
| ----------------------------- | ------ | ----------------------------------------------------------- |
| Log line buffer               | ~20MB  | 10,000 lines, ~2KB average parsed representation            |
| Tree state                    | ~5MB   | Full `RootIndex` plus cached `NodeState` for expanded nodes |
| Viewport buffers              | ~10MB  | Rendered content for visible panes                          |
| Unblock session history       | ~5MB   | Conversation history, capped at 200 messages                |
| Overhead (runtime, libraries) | ~10MB  | Bubbletea, lipgloss, fsnotify                               |

### 7.2 Lazy Loading

Trees with more than 100 nodes use lazy loading: only expanded subtrees have their `NodeState` loaded from disk. Collapsing a subtree releases its cached state after a 30-second grace period (in case the user re-expands).

The multi-engineer view (v0.6) loads namespace trees on demand. Only the visible portion of each namespace tree is fully loaded.

### 7.3 Virtual Scrolling

The log stream viewport and any list with more than 500 items use virtual scrolling: only the visible rows plus a small overscan buffer (20 rows above and below) are rendered. The `View()` method calculates the visible window from the scroll offset and total item count, renders only those items, and pads the rest with empty lines.

### 7.4 Debouncing and Batching

File system events are debounced at 100ms (described in section 5.4). View re-renders are additionally batched: if multiple `tea.Msg` events arrive within the same Bubbletea tick, only one `View()` call fires.

### 7.5 Startup Time

Target: the TUI should be interactive within 200ms of launch. Startup reads only the root index and PID file. Node states, log files, and other data load asynchronously after the first render.

### 7.6 Large Trees (1000+ Nodes)

Trees beyond a few hundred nodes require attention in three areas:

**Aggregate counts:** Header line 2 shows node counts by status. Recomputing these by iterating all `IndexEntry` values on every `StateUpdatedMsg` is O(n). For trees over 500 nodes, the TUI maintains incremental counters: each `NodeUpdatedMsg` adjusts the delta (old status -1, new status +1) rather than rescanning.

**Position indicator:** When the tree contains more nodes than fit on screen, the tree pane header shows a scroll position indicator: `47/1203` (current cursor index / total visible nodes). This serves the same role as a scrollbar thumb without requiring mouse interaction.

**fsnotify watch limits:** Each `NodeState` file gets its own fsnotify watch. On Linux, `inotify` has a per-user watch limit (default 8192 on most distributions, configurable via `fs.inotify.max_user_watches`). For trees over 1000 nodes, the TUI watches only the root index and the currently expanded subtree's node states, not every node file. Collapsed subtrees rely on the polling fallback (section 5.2). On macOS, kqueue has no practical limit for this scale.

### 8.4 Accessibility (v0.1+)

**Screen readers:** Bubbletea renders to a raw terminal, which screen readers cannot reliably interpret. The TUI is a visual tool and does not claim screen reader support. Users who need screen-reader-compatible output should use `wolfcastle status --json` and `wolfcastle follow`, which emit structured text to stdout.

**High-contrast theme (v0.1):** A built-in `high-contrast` theme is available via `tui.theme_preset: "high-contrast"` in `local/config.json`. It uses pure white on pure black, with bright primary colors for status glyphs. No color pairing relies solely on hue to convey meaning.

**Color blindness (v0.1):** Status glyphs use distinct shapes (filled circle, half circle, empty circle, trefoil) in addition to color. The shape alone is sufficient to distinguish the four states. Audit status glyphs follow the same principle (pause, half circle, empty circle, slashed circle). Users who find the default glyphs hard to distinguish can override them via `tui.glyphs` (see section 8.3).

### 10.1 Distribution

The TUI is included in the default binary. Headless environments (CI, servers) can opt out.

**GoReleaser** produces two binaries per platform:
- `wolfcastle` (default): includes TUI
- `wolfcastle-headless`: built with `-tags headless`, no Charm dependencies

**Homebrew:**
- `brew install dorkusprime/tap/wolfcastle` installs the full binary (TUI included)
- `brew install dorkusprime/tap/wolfcastle-headless` installs the lean binary

**curl installer:**
- `curl -sSL ... | sh` installs the full binary by default
- `curl -sSL ... | sh -s -- --headless` installs without TUI

The primary user is a developer on a local machine. The TUI is the expected experience. Headless is for CI pipelines, remote servers, and Docker containers where terminal rendering is irrelevant.

### 11.7 Error Display

Errors fall into two categories:

**Transient errors** (network blip, file temporarily locked) appear as a toast: `State file unreadable. Retrying.` The TUI shows the last known good state and retries on the next poll tick.

**Persistent errors** (corrupt JSON, schema version mismatch, missing required files) trigger a sticky error bar above the footer. This bar does not auto-dismiss. It reads: `State corruption detected: {filename}. Run wolfcastle doctor.` The bar remains until the file is successfully re-read or the user presses `Esc` to dismiss. If multiple files are corrupt, the bar stacks messages (up to 3 visible; a count badge shows any overflow).

The doctor suggestion is the primary recovery path. The TUI does not attempt automatic repair. If the user runs `wolfcastle doctor` in another terminal and it fixes the file, the next fsnotify event or poll tick clears the error bar automatically.

## 12. Open Questions

These remain unresolved and need answers before the spec is accepted.

1. **Should the TUI replace `follow` and `status`?** The TUI subsumes their functionality. Do we deprecate them, keep them as lightweight alternatives, or alias them into `tui` subcommands?
2. **OSC 52 clipboard support coverage.** OSC 52 works in iTerm2, kitty, WezTerm, and recent Alacritty. It does not work in Apple Terminal or older tmux versions without `set -g set-clipboard on`. Do we need a fallback (e.g., `pbcopy`/`xclip` detection), or is documenting the limitation sufficient?
3. **Log buffer vs. disk for long-running sessions.** The 10,000-line in-memory log buffer means old lines are lost. Should the TUI support scrolling back into the on-disk log file (lazy loading from the `.jsonl` file by seeking backward)?
4. **fsnotify reliability on NFS/FUSE mounts.** The polling fallback exists, but the poll interval (2s) may feel sluggish. Should the TUI detect non-native filesystems and automatically lower the poll interval?
5. **Multi-engineer write safety.** The TUI writes to the active engineer namespace. If multi-engineer view is open showing namespace B while the active namespace is A, is the visual context misleading enough to cause accidental writes to the wrong namespace?

---

## 13. Permanent Exclusions

These are out of scope for all planned versions. They represent architectural boundaries, not deferred work.

- **Mouse support.** Keyboard only. Mouse support in Bubbletea is possible but adds interaction complexity for little gain in a terminal tool.
- **Remote daemon observation.** The TUI reads local files only. No network transport, no SSH tunneling, no daemon API.
- **Log file compression awareness.** The TUI reads `.jsonl` files only. It does not decompress `.jsonl.gz` files for historical browsing. If you need old logs, use `zcat`.
- **Daemon lifecycle control.** No start, stop, restart, or pause from the TUI. Those are separate CLI commands and should stay that way. The TUI observes the daemon; it does not own it.
