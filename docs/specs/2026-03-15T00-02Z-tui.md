# Wolfcastle TUI

A terminal user interface for observing and managing the Wolfcastle daemon, built on Bubbletea.

The `wolfcastle follow` and `wolfcastle status` commands work, but they are separate views into the same system. The TUI combines them into a single, persistent interface: a live dashboard where you can watch the daemon hunt, browse the project tree, read logs, and manage the inbox without leaving the terminal.

---

## 1. Purpose and Goals

The TUI is a read-mostly observation tool. Its job is to answer three questions at a glance:

- **What is the daemon doing right now?** (Dashboard, log stream)
- **What does the project tree look like?** (Tree browser, node/task detail)
- **What needs my attention?** (Blocked tasks, open gaps, escalations, inbox)

It is not a control plane. v1 does not start, stop, or configure the daemon. It does not mutate state files. It watches, filters, and presents.

## 2. Layout and Panes

The interface uses a split-pane layout with a header bar, two main content areas, and a footer.

```
┌─────────────────────────────────────────────────────────────────┐
│ WOLFCASTLE v0.1.0                      Daemon: hunting (PID X)  │
│ 12 nodes: 4● 3◐ 3○ 2✕   Audit: 2 passed, 1 gap open          │
├──────────────────────┬──────────────────────────────────────────┤
│                      │                                          │
│   Project Tree       │   Detail Pane                            │
│                      │                                          │
│ ▸ auth-system    ●   │   (varies by context: node detail,       │
│   ▸ login     ◐     │    task detail, log stream, inbox)       │
│   ▸ oauth     ○     │                                          │
│ ▸ api-gateway    ●   │                                          │
│   ▸ rate-limit ✕    │                                          │
│                      │                                          │
├──────────────────────┴──────────────────────────────────────────┤
│ [q]uit  [d]ashboard  [l]ogs  [i]nbox  [/]search  [?]help       │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 Header Bar

Two lines, always visible.

**Line 1:** Product name, version, daemon status. The daemon status reads the PID file and probes the process, same as `getDaemonStatus` does today.

**Line 2:** Aggregate counts. Node status breakdown using the four-glyph vocabulary (see 2.5). Audit summary: counts by AuditStatus, plus open gap and escalation totals.

### 2.2 Left Pane: Project Tree

A navigable tree built from the `RootIndex`. Each entry shows the node name and a status glyph. Orchestrator nodes are collapsible; expanding one reveals its children. Leaf nodes show their tasks when expanded.

The tree respects the index's `Root` ordering for top-level nodes and `Children` ordering within orchestrators.

Width: fixed at roughly 30% of the terminal, with a minimum of 24 columns. If the terminal is too narrow (under 60 columns), the tree pane collapses and the detail pane fills the screen; pressing `t` toggles the tree back.

### 2.3 Right Pane: Detail

The detail pane's content depends on what's selected and which mode is active:

| Mode | Trigger | Content |
|------|---------|---------|
| Dashboard | `d` key or startup default | Iteration progress, daemon health, recent activity summary |
| Node detail | Enter on an orchestrator | Node name, type, status, children summary, audit state, breadcrumbs, gaps, escalations |
| Task detail | Enter on a task | Task ID, title, description, status, failure count, block reason, breadcrumbs, `is_audit` flag, `needs_decomposition` flag |
| Log stream | `l` key | Live NDJSON log output, formatted and colorized |
| Inbox | `i` key | Inbox items list with status, text input for adding new items |

The detail pane is a `bubbles/viewport` (scrollable). In log stream mode it supports both follow mode (auto-scroll to bottom) and manual scrolling.

### 2.4 Footer

A single line showing available key bindings for the current context. The bindings change depending on which pane has focus and which mode the detail pane is in.

### 2.5 Status Glyphs

| Glyph | Status | Color |
|-------|--------|-------|
| `●` | complete | green |
| `◐` | in_progress | yellow |
| `○` | not_started | dim white |
| `✕` | blocked | red |

Audit status uses the same position but different glyphs when the audit view is active:

| Glyph | Audit Status |
|-------|-------------|
| `✓` | passed |
| `◐` | in_progress |
| `·` | pending |
| `✗` | failed |

## 4. Data Sources

Each pane reads from specific files and APIs. All reads go through the existing `StateStore` or direct file I/O. No state mutations in v1 except inbox additions.

### 4.1 Header Bar

- **Daemon status:** Read `$WOLFCASTLE_DIR/wolfcastle.pid`, parse PID, probe with `signal(0)`. Same logic as `getDaemonStatus`.
- **Aggregate counts:** Derived from the `RootIndex` (node status counts) and a pass over leaf `NodeState` files (audit counts, open gaps, open escalations).

### 4.2 Project Tree

- **Tree structure:** `StateStore.ReadIndex()` returns the `RootIndex`. The `Nodes` map provides every node's name, type, status, parent, and children. The `Root` slice gives top-level ordering.
- **Task list (leaf expansion):** `StateStore.ReadNode(addr)` returns the `NodeState` with its `Tasks` slice.

### 4.3 Node / Task Detail

- **Node detail:** `StateStore.ReadNode(addr)` for the full `NodeState`, including `Audit` (scope, breadcrumbs, gaps, escalations), `Children`, and `Tasks`.
- **Task detail:** Comes from the parent `NodeState.Tasks` slice, filtered by task ID.

### 4.4 Log Stream

- **Log files:** Read from `$WOLFCASTLE_DIR/logs/*.jsonl`. Use `logging.LatestLogFile` to find the current file, then tail it.
- **Parsing:** Each line is a JSON object. The TUI parses the same fields as `formatAndPrintLogLine`: `type`, `level`, `trace`, `stage`, `message`, `text`, `node`, `task`, `exit_code`, `error`, `count`, `reason`.
- **Formatting:** Colorized by level (debug=dim, info=white, warn=yellow, error=red). Trace IDs shown as prefixes: `[exec-0042]`, `[intake-10003]`.
- **Historical lines:** On first load, show the last 50 lines from the current log file. Then stream new lines as they appear.

### 4.5 Inbox

- **Read:** `StateStore.ReadInbox()` returns `InboxFile` with its `Items` slice.
- **Write (add item):** `StateStore.MutateInbox()` appends a new `InboxItem` with timestamp, text, and status `"new"`. This is the single mutation the TUI performs.

## 5. Real-Time Update Strategy

The TUI needs to reflect changes as the daemon works. Two mechanisms, chosen per data type:

### 5.1 File Watching with fsnotify

Primary mechanism. Watch these paths:

| Path | Triggers update of |
|------|--------------------|
| `$WOLFCASTLE_DIR/wolfcastle.pid` | Header daemon status |
| `$PROJECTS_DIR/state.json` (root index) | Tree structure, header counts |
| `$PROJECTS_DIR/*/state.json` (node states) | Node detail, task detail, audit info |
| `$PROJECTS_DIR/inbox.json` | Inbox pane |
| `$WOLFCASTLE_DIR/logs/*.jsonl` | Log stream (new data in current file) |

When fsnotify fires a write event, the TUI sends a `tea.Msg` into the Bubbletea event loop. The relevant sub-model re-reads its data and the view re-renders.

For the log directory, also watch for new file creation events. When a new `.jsonl` file appears (new iteration), switch the log stream to tail it.

### 5.2 Polling Fallback

fsnotify does not work on all filesystems (notably some network mounts). The TUI also runs a low-frequency poll (every 2 seconds) that re-checks the root index and PID file. If fsnotify is delivering events, the poll is a no-op (compare file mod times before re-reading).

### 5.3 Log Tailing

The log stream deserves special treatment because it updates at a higher frequency than tree state.

The TUI maintains a file offset (same approach as `tailFileStreaming` in `follow.go`). On each fsnotify event or poll tick, it seeks to the offset, reads new lines, parses and appends them to the viewport buffer, and updates the offset.

When the daemon starts a new iteration (detected by a new file appearing in the logs directory), the TUI prints a separator line and switches to the new file.

Buffer size: keep the last 10,000 parsed log lines in memory. Older lines are discarded. This prevents unbounded memory growth during long daemon runs.

### 5.4 Update Debouncing

State file changes can come in bursts (the daemon writes the node state, then the index, then propagates to parents, all within milliseconds). The TUI debounces fsnotify events with a 100ms window: collect all events in the window, then re-read only the affected files once.

## 6. Bubbletea Architecture

### 6.1 Model Composition

The top-level model owns the layout and delegates to sub-models for each pane:

```
TUIModel
├── HeaderModel        (daemon status, aggregate counts)
├── TreeModel          (project tree navigation)
├── DetailModel        (container that swaps between:)
│   ├── DashboardModel   (iteration progress, summary)
│   ├── NodeDetailModel  (single node state + audit)
│   ├── TaskDetailModel  (single task detail)
│   ├── LogViewModel     (streaming log output)
│   └── InboxModel       (inbox list + text input)
├── FooterModel        (context-sensitive key hints)
└── HelpOverlay        (toggleable help screen)
```

### 6.2 Message Types

Custom `tea.Msg` types for data updates:

- `StateUpdatedMsg` carries a fresh `RootIndex` and triggers tree/header re-render.
- `NodeUpdatedMsg` carries a fresh `NodeState` for a specific address.
- `LogLinesMsg` carries a batch of new parsed log lines.
- `DaemonStatusMsg` carries the daemon status string.
- `InboxUpdatedMsg` carries a fresh `InboxFile`.
- `NewLogFileMsg` signals iteration rollover with the new file path.

### 6.3 Subscriptions

A single `tea.Sub` manages the fsnotify watcher and polling loop. It sends the appropriate message types into the Bubbletea event loop. The subscription is started once in `Init()` and runs for the lifetime of the TUI.

The subscription goroutine:
1. Sets up fsnotify watches on the relevant paths.
2. Enters a select loop on fsnotify events, poll timer, and a done channel.
3. On each event, reads the affected file(s) and sends a typed message via the program's `Send` method.

### 6.4 Focus Management

The `TUIModel` tracks which pane has focus (`tree` or `detail`). Key events route to the focused pane's `Update` method first. Global keys are handled at the top level before delegation.

The `Tab` key cycles focus. Visual indicator: the focused pane gets a highlighted border (lipgloss border style swap).

### 6.5 Responsive Sizing

`tea.WindowSizeMsg` triggers re-layout. The `TUIModel` recalculates column widths (tree: 30%, detail: 70%, respecting minimums) and propagates new dimensions to sub-models. Each sub-model's `View()` respects its allocated width and height.

Below 60 columns total width, the tree pane hides and the detail pane goes full-width.

## 7. Package Structure

```
cmd/tui/
    main.go          Entry point: parse flags, create StateStore, start Bubbletea program
    model.go         Top-level TUIModel, Init, Update, View, layout logic
    keys.go          Key binding definitions (key.Binding from bubbles/key)
    messages.go      Custom tea.Msg types
    watcher.go       fsnotify setup, polling loop, subscription function
    styles.go        lipgloss style definitions (colors, borders, glyphs)

cmd/tui/header/
    model.go         HeaderModel: daemon status, aggregate counts

cmd/tui/tree/
    model.go         TreeModel: tree data structure, cursor, expand/collapse
    render.go        Tree rendering with indentation and glyphs

cmd/tui/detail/
    model.go         DetailModel: mode switching, delegates to sub-views
    dashboard.go     DashboardModel: iteration summary, recent activity
    nodedetail.go    NodeDetailModel: node state, audit trail
    taskdetail.go    TaskDetailModel: single task view
    logview.go       LogViewModel: scrollable log output, filtering, follow mode
    inbox.go         InboxModel: item list, text input for new items

cmd/tui/help/
    model.go         HelpOverlay: centered overlay with grouped key bindings
```

The TUI is a new `cobra.Command` registered under the root: `wolfcastle tui`. It is separate from the existing `follow` and `status` commands, which remain as quick non-interactive alternatives.

## 9. Voice and Personality

The TUI follows the Wolfcastle voice guide. The interface is terse, confident, and slightly menacing. A few specific applications:

### 9.1 Daemon Status (Header)

| Actual state | Display text |
|-------------|-------------|
| Running | `Daemon: hunting (PID 12345)` |
| Stopped (clean) | `Daemon: standing down` |
| Stopped (stale PID) | `Daemon: presumed dead (stale PID 12345)` |
| Unknown | `Daemon: status unknown` |

### 9.2 Empty States

| Context | Text |
|---------|------|
| No tree (no root index) | `No targets. Feed the inbox.` |
| All nodes complete | `All targets eliminated.` |
| All nodes blocked | `Blocked on all fronts. Human intervention required.` |
| No log files | `No transmissions. The daemon has not spoken.` |
| Empty inbox | `Inbox empty. The silence is temporary.` |

### 9.3 Tree Expansion (no children/tasks)

When expanding a node that has no children or tasks yet: `Awaiting decomposition.`

### 9.4 Log Level Filter Indicator

The current log level filter displays in the log pane header as: `Level: INFO and above` or `Level: all (unfiltered)`.

### 9.5 Error Display

Errors in the TUI itself (failed to read state file, fsnotify error) appear as a brief flash message in the footer: `State file unreadable. Retrying.` The TUI does not crash on transient read errors; it shows the last known good state and retries on the next poll.

## 10. What v1 Does Not Build

These are explicitly out of scope. They may come later, but the first version is an observation tool.

- **Daemon control.** No start, stop, restart, or pause from the TUI. Use the existing CLI commands.
- **State mutation (beyond inbox).** No marking tasks complete, unblocking nodes, or editing descriptions. The TUI reads state; the daemon writes it.
- **Configuration editing.** No editing wolfcastle.yaml or tier configs from the TUI.
- **Worktree management.** No creating, listing, or deleting worktrees.
- **Multiple namespace support.** v1 shows the active engineer namespace only. The `--all` flag from `status` is not replicated.
- **Mouse support.** Keyboard only. Mouse support in Bubbletea is possible but adds interaction complexity for little gain in a terminal tool.
- **Theming or custom color schemes.** One color scheme. It should look good on dark terminals. Light terminal users can file a complaint.
- **Remote daemon observation.** The TUI reads local files only. No network transport, no SSH tunneling, no daemon API.
- **Log file compression awareness.** The TUI reads `.jsonl` files only. It does not decompress `.jsonl.gz` files for historical browsing. If you need old logs, use `zcat`.
