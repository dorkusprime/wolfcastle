# TUI Tabs: Per-Directory Workspaces

## Status

Accepted

## Summary

Replace the instance-list tab bar with user-managed tabs. Each tab owns a directory, an independent store, watcher, daemon lifecycle, and detail pane. Tabs can target any directory on disk, whether it has a running daemon, an initialized `.wolfcastle/`, or nothing at all. Uninitialized directories prompt for init.

The current model ties the TUI to whatever instances the registry happens to contain. That works when daemons are already running but makes it impossible to open, inspect, or start work in a directory that has no active daemon. Tabs remove that limitation.

## Keybindings

| Key | Action |
|-----|--------|
| `+` | Open a new tab (directory picker overlay) |
| `-` | Close the current tab (confirmation if daemon running) |
| `<` | Switch to previous tab |
| `>` | Switch to next tab |

`<` and `>` already exist for instance switching. This spec repurposes them for tab switching (tabs subsume instances).

Tab switching is blocked while a modal overlay (log, inbox, daemon confirm) is open. Dismiss the modal first, then switch. This prevents confusing state where a modal from one tab renders over another tab's content.

## Tab Model

Each tab is a self-contained workspace:

```go
type Tab struct {
    ID          int                  // stable identifier, never reused
    Label       string               // display name (basename of directory)
    WorktreeDir string               // absolute path to the directory
    Store       *state.Store         // nil until .wolfcastle/ is resolved
    DaemonRepo  *daemon.Repository   // nil until .wolfcastle/ is resolved
    Watcher     *tui.Watcher         // nil until Start(); stopped on Close()
    Events      chan tea.Msg          // per-tab watcher channel
    EntryState  EntryState           // Cold, Live, Welcome, Browse
    Tree        tree.Model
    Detail      detail.Model
    Search      search.Model         // per-tab search query and highlights
    PrevIndex   *state.RootIndex     // for diff/notification
    PrevNodes   map[string]*state.NodeState
    Focused     FocusedPane          // which pane has focus (tree vs detail)
    LastFocused FocusedPane          // for restoring after modal dismiss
    TreeVisible bool                 // whether the tree pane is shown
    Loading     bool                 // true while initial state load is in flight
    DaemonStarting bool
    DaemonStopping bool
    Errors      []errorEntry         // directory-specific errors
}
```

Tabs are identified by a stable `ID` (monotonically increasing counter), not by slice index. All in-flight messages (`TabMsg`, poll ticks) reference the tab ID, so closing a tab and shifting the slice doesn't cause misrouted messages.

The outer `TUIModel` holds `tabs []Tab`, `activeTabID int`, and `nextTabID int`. Shared state (header, footer, help overlay, notification toasts, activeModal, daemonModal, instances, originalCWD, version) stays on `TUIModel`.

### Entry States

| State | Meaning | What renders |
|-------|---------|-------------|
| `Welcome` | No directory selected yet (only on first launch before any tab exists) | Welcome screen with directory browser and running sessions |
| `Cold` | `.wolfcastle/` exists, no daemon running | Tree and dashboard, `s` starts daemon |
| `Live` | `.wolfcastle/` exists, daemon running | Tree and dashboard with live updates, `s` stops daemon |
| `Browse` | Directory selected but no `.wolfcastle/`, user declined init | Directory listing, `s` triggers init prompt |

### Tab Lifecycle

**Creation (`+`)**

1. A directory picker overlay appears (similar to the welcome screen's directory browser, rooted at CWD or home). A "Running sessions" section at the top lists instances from the registry.
2. The user navigates to a directory or selects a running session and presses Enter.
3. Duplicate check: if a tab already exists for that directory, switch to it instead of creating a new one. (Non-goal says "duplicates refocus"; this is where it's enforced.)
4. A new tab is created with a fresh stable ID.
5. If `.wolfcastle/` exists: the tab enters Cold or Live state depending on whether a daemon is registered. `Tab.Start()` is called.
6. If `.wolfcastle/` does not exist: the tab shows a prompt: "No project here. Initialize? [y/n]". On `y`, runs `wolfcastle init` in that directory, then enters Cold state and calls `Tab.Start()`. On `n`, the tab enters Browse state (directory listing, no watcher, no poller).
7. If `.wolfcastle/` exists but the base tier is missing (fresh git clone): regenerate tiers automatically (same as `wolfcastle start` already does), then enter Cold state.
8. The new tab becomes the active tab. Any open modal is dismissed.

**Switching (`<` / `>`)**

Blocked while a modal is open (user must dismiss first). Rotates `activeTabID` through the tab list. The header tab bar updates to highlight the new active tab. The split-pane layout swaps to the active tab's Tree, Detail, Search, and focus state. No re-render of background tabs.

**Closing (`-`)**

1. Cannot close the last tab. If only one tab exists, `-` is a no-op (or shows a toast: "Last tab. Use `q` to quit.").
2. If the tab has a running daemon, show a confirmation: "Daemon running in <dir>. Stop before closing? [y/n/Esc]". `y` stops the daemon and closes the tab. `n` closes the tab without stopping (daemon keeps running; can reconnect later via `+`). `Esc` cancels.
3. `Tab.Stop()` is called: watcher stopped, channel drained, resources released.
4. Tab removed from `tabs` slice.
5. Adjacent tab becomes active (prefer next; if closing the last tab in the list, select previous).

**Tab.Start()**

Called when a tab has a resolved `.wolfcastle/` directory. Returns a `tea.Cmd` batch:

1. Creates a `tui.Watcher` with the tab's store and daemon repo, pointed at the tab's `Events` channel.
2. Calls `watcher.Start()` (best-effort fsnotify) and `watcher.StartPolling()` (always, for reliability).
3. Calls `watcher.EagerPrefetchAndSubscribe()` to populate the tree cache.
4. Starts the per-tab poll chain: `tea.Tick(2s)` that emits `TabPollTickMsg{TabID: id}`.
5. Starts the per-tab event drain: a Cmd that blocks on `<-tab.Events` and wraps the result in `TabMsg{TabID: id, Inner: msg}`.
6. Loads initial state (`store.ReadIndex()`) and emits it as a `TabMsg`.

**Tab.Stop()**

1. Stops the watcher (`watcher.Stop()`).
2. Drains the `Events` channel.
3. The poll chain stops naturally: when the Update loop receives a `TabPollTickMsg` for a tab ID that no longer exists, it does not reschedule.
4. Same for the event drain: a `TabMsg` for a dead tab ID is silently dropped.

### First Launch

When `wolfcastle` launches:

1. If CWD has `.wolfcastle/`: create one tab for CWD. Call `Tab.Start()`.
2. If CWD has no `.wolfcastle/` but instances exist in the registry: create one tab for the CWD-matching instance (longest-prefix match, same as today's `Resolve`). If no CWD match, create one tab for the first instance. Additional instances are discoverable via `+`.
3. If neither: create one tab in Welcome state. The welcome screen's "Connect to session" and "Initialize here" actions replace this tab's state (Welcome to Live or Cold).

Case 2 creates exactly one tab, not one per instance. Opening tabs for every running daemon would be surprising. The user adds more via `+`.

### Instance Discovery Integration

The instance registry remains. A global poll (on `TUIModel`, not per-tab) checks the registry directory mtime every 2 seconds. When the registry changes:

- For each tab: if a daemon started or stopped in that tab's worktree, update the tab's `EntryState` (Cold to Live, or Live to Cold). Emit a toast notification.
- New daemons that don't match any open tab are not auto-opened (the user adds them via `+`).
- Stale instances (dead PIDs) are pruned as today.

## Header Tab Bar

The header's third line (currently the instance tab bar, shown when width > 100) becomes the tab bar:

```
[wc-tui-test ●] [my-saas-app] [+]
```

- Active tab: bold, with `●` if daemon is running in that tab's directory.
- Inactive tabs: dim, with `●` if their daemon is running (so the user can see which tabs have active work).
- `[+]` at the end: visual hint for the `+` key.
- Overflow: when tabs exceed the header width, show `‹ N` / `N ›` indicators (not `<`/`>`, which would be confused with the tab-switch keybindings) and scroll the visible window to keep the active tab in view.

When the terminal is narrower than 100 columns, the tab bar is hidden but `<`/`>` still work. The active tab label is shown in the first header line alongside the version string.

## Daemon Lifecycle Scoping

All daemon commands (`s` to start/stop, the daemon modal) operate on the **active tab's directory**. `startDaemon` uses `tabs[activeTab].WorktreeDir`. `stopCurrentDaemon` resolves the instance for `tabs[activeTab].WorktreeDir`. The daemon modal's confirmation text includes the directory name so the user knows which daemon they're affecting.

If the active tab is in Browse state (no `.wolfcastle/`), pressing `s` triggers the init prompt. On confirmation, the tab initializes, enters Cold, calls `Tab.Start()`, then proceeds to start the daemon.

## Detail Pane Scoping

Each tab owns its own `detail.Model`. When switching tabs, the detail pane content swaps entirely. The detail mode (Dashboard, NodeDetail, TaskDetail, LogStream, Inbox) is per-tab. Opening the log modal in one tab doesn't affect another tab's detail state.

## Watcher Scoping

Each tab has its own `tui.Watcher` and `chan tea.Msg`. The watcher emits to its tab's channel. A per-tab `waitForTabEvent` Cmd drains the channel and wraps events in `TabMsg{TabID, Inner}`. The Update loop dispatches `TabMsg` to the correct tab by ID.

Watchers for inactive tabs continue running, and their drain Cmds continue delivering `TabMsg` to the Update loop. The Update loop updates the background tab's state (tree, detail, prevIndex) but does not trigger a re-render unless the tab is active.

Polling always runs alongside fsnotify (belt and suspenders). Per-node mtime polling is included in every tab's `pollTick`.

## Search

Search (`/`) operates on the active tab's tree. The search model is per-tab: each tab can have an independent search query and highlight state. Switching tabs swaps the search UI to the new tab's query. The search overlay itself (the input bar at the bottom) is rendered by the active tab's search model.

## Notifications

Notification toasts are global. A toast from any tab is visible regardless of which tab is active. The toast text includes the tab label when the source tab is not the active one (e.g., "wc-tui-test: New target acquired: auth/login").

## Persistence

Tab state is not persisted across TUI sessions. Each launch starts fresh from CWD/instance-discovery. A future enhancement could save tab layout, but that's out of scope for this spec.

## Architectural Invariant

**Every piece of directory-dependent state lives on `Tab`, never on `TUIModel`.**

The bugs we've been fixing (daemon starting in CWD, stale tree after instance switch, logs showing "No transmissions," poll chain not running) all stem from one structural problem: `TUIModel` holds `worktreeDir`, `store`, `daemonRepo`, `watcher`, `entryState`, `prevIndex`, and `prevNodes` as top-level fields. When the TUI switches contexts, each of those fields must be updated in lockstep. Miss one and something breaks silently.

Tabs eliminate the coordination problem by making the boundary physical. A `Tab` is the unit of directory context. The `TUIModel` never holds a store, watcher, worktreeDir, or entryState directly. Every function that needs directory context receives it from the active tab, not from a top-level field that might be stale.

| Field | Reason |
|-------|--------|
| `width`, `height` | Terminal dimensions, shared |

### Message Routing

Every `tea.Msg` that carries directory-scoped data must be tagged with a stable tab ID:

```go
type TabMsg struct {
    TabID int
    Inner tea.Msg
}

type TabPollTickMsg struct {
    TabID int
}
```

The Update loop dispatches by ID, not index:

```go
case TabMsg:
    tab := m.tabByID(msg.TabID)
    if tab == nil {
        return m, nil // tab was closed, drop silently
    }
    // update tab's tree, detail, prevIndex, etc.
    // reschedule drain for this tab

case TabPollTickMsg:
    tab := m.tabByID(msg.TabID)
    if tab == nil {
        return m, nil // tab closed, poll chain dies naturally
    }
    // re-read index, reschedule tick
```

Messages from the active tab trigger re-render. Messages from background tabs update state silently.

### What This Eliminates

| Bug class | Root cause | Why tabs fix it |
|-----------|-----------|----------------|
| Daemon starts in CWD | `m.worktreeDir` is CWD before switch | `startDaemon` reads from active tab's `WorktreeDir`, set at tab creation |
| Tree freezes after switch | Poll chain not started (store was nil at Init) | Every tab starts its own poller at creation |
| Logs show "No transmissions" | Watcher events not drained (drain not scheduled at Init) | Every tab starts its own drain at creation |
| Stale task data | Per-node polling missing, fsnotify silent | Per-tab watcher always runs polling alongside fsnotify |
| State mismatch after switch | Multiple fields updated non-atomically | All directory state is a single `Tab` value; switching is just changing `activeTabID` |
| Misrouted messages after close | Index-based routing shifts when tabs close | Stable ID routing; dead IDs are silently dropped |

## Migration from Instance Tabs

The current `m.instances`, `m.activeInstanceIndex`, `reconcileActiveInstance`, and `switchInstance` are removed. Instance discovery still runs (global poll on the registry directory), but it updates existing tabs' `EntryState` rather than driving tab creation. The `<>` keybindings switch tabs instead of instances.

The `InstanceSwitchedMsg`, `WorktreeGoneMsg`, and the entire `instance_reconcile.go` file are replaced by tab lifecycle methods. `DaemonStatusMsg` updates the active tab's entry state directly.

Breaking changes: none user-facing. The tab bar looks similar but is now user-managed rather than registry-driven.

## Implementation Phases

### Phase 1: Tab Infrastructure

- Define `Tab` struct with stable ID
- Move per-directory state from `TUIModel` to `Tab`
- `tabByID` helper for message routing
- `TabMsg` and `TabPollTickMsg` message types
- `Tab.Start()` and `Tab.Stop()` lifecycle methods
- Tab creation from current initialization paths (CWD, instance discovery)
- `<` / `>` switch between tabs (blocked during modals)
- Header tab bar renders from `[]Tab`
- Single-tab behavior is identical to current behavior
- Remove `reconcileActiveInstance`, `switchInstance`, `InstanceSwitchedMsg`
- Instance discovery updates existing tabs' EntryState

### Phase 2: New Tab (`+`)

- Directory picker overlay (reuse/extend welcome screen's directory browser)
- "Running sessions" section in picker (from instance registry)
- Duplicate detection (refocus existing tab)
- Init prompt for uninitialized directories
- Tier regeneration for partial scaffolds
- Browse state for declined init
- `s` in Browse state triggers init prompt

### Phase 3: Close Tab (`-`)

- Daemon-running confirmation with directory name
- `Tab.Stop()` cleanup
- Last-tab guard (no-op or toast)
- Adjacent-tab selection after close

### Phase 4: Polish

- Tab overflow scrolling with `‹ N` / `N ›` indicators
- Narrow terminal: active tab label in header line 1
- Cross-tab toast attribution ("wc-tui-test: New target acquired")
- Welcome-to-tab transition (replace Welcome state with real tab)

## Non-Goals

- Tab reordering. Use `<>` to navigate.
- Tab persistence across sessions.
- Multiple views of the same directory. Duplicates refocus the existing tab.
- Remote directories or SSH-tunneled projects.
- Mouse click on tab bar to switch (Bubbletea v2 has mouse support, but it's fragile across terminals; defer to a future spec).
