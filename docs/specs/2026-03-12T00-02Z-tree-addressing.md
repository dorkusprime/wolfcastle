# Tree Addressing Scheme

## Overview

Every node in a Wolfcastle project tree has a unique address. This address is a slash-separated path from the tree root to the target node. All CLI commands that operate on nodes accept a tree address via the `--node` flag (ADR-008). This spec defines the address format, how addresses map to a distributed set of co-located files on disk (ADR-024), how the Go code resolves them, and the rules for valid and invalid addresses.

## Address Format

A tree address is a slash-separated sequence of node slugs:

```
<slug>[/<slug>[/<slug>[...]]]
```

Each slug is a kebab-case identifier: lowercase ASCII letters, digits, and hyphens. No leading or trailing hyphens. No consecutive hyphens. No empty segments. No leading or trailing slashes.

### Grammar

```
address        = slug *("/" slug)
slug           = ALPHA *(ALPHA / DIGIT / "-")
ALPHA          = %x61-7A           ; a-z
DIGIT          = %x30-39           ; 0-9
```

A slug must start with a letter and must not end with a hyphen.

### Depth Levels

Addresses have no fixed maximum depth, but typical trees are 2-4 levels deep:

| Depth | Role | Example |
|-------|------|---------|
| 1 | Project (root orchestrator) | `attunement-tree` |
| 2 | Sub-project (orchestrator or leaf) | `attunement-tree/fire-impl` |
| 3 | Leaf node or further nesting | `attunement-tree/fire-impl/stamina-costs` |
| N | Task within a leaf | `attunement-tree/fire-impl/stamina-costs/task-3` |

The final segment in a task address is a task identifier, not a node slug. Task identifiers are auto-generated by `wolfcastle task add` and follow the format `task-N` where N is a monotonically increasing integer within that leaf.

## Node Types

The tree contains two types of nodes:

1. **Orchestrator nodes** contain sub-projects (child nodes). They do not contain tasks directly. Their job is to coordinate and audit the work of their children.

2. **Leaf nodes** contain tasks. They do not contain sub-projects. Work happens here.

The node type is determined by its contents in the node's `state.json`, not by its depth in the tree. A depth-1 node can be a leaf (a simple project with tasks and no sub-projects). A depth-3 node can be an orchestrator (deeply nested coordination).

## Examples

Throughout this spec, the following example tree is used. It represents an engineer `wild-macbook` working on a game's attunement system:

```
attunement-tree                          # orchestrator (root)
  attunement-tree/fire-impl              # leaf
  attunement-tree/water-impl             # leaf
  attunement-tree/balance-pass           # orchestrator
    attunement-tree/balance-pass/pvp     # leaf
    attunement-tree/balance-pass/pve     # leaf
```

### Addressing at Every Level

**Root project:**
```
wolfcastle status --node attunement-tree
```

**Sub-project (leaf):**
```
wolfcastle task add --node attunement-tree/fire-impl "Wire stamina cost to fire spells"
```

**Sub-project (orchestrator):**
```
wolfcastle project create --node attunement-tree/balance-pass "pvp"
```

**Leaf inside nested orchestrator:**
```
wolfcastle task claim --node attunement-tree/balance-pass/pvp
```

**Task within a leaf:**
```
wolfcastle task complete --node attunement-tree/fire-impl/task-2
```

## Filesystem Mapping

### Engineer Namespace

All live project state lives under the engineer's namespace directory (ADR-009):

```
.wolfcastle/system/projects/{user}-{machine}/
```

The identity is resolved from `local/config.json`:

```json
{
  "identity": {
    "user": "wild",
    "machine": "macbook"
  }
}
```

This produces the root path `.wolfcastle/system/projects/wild-macbook/`.

### Distributed State: Per-Node `state.json` Files

State is distributed as one `state.json` per node, co-located with that node's project description and task documents (ADR-024). Each node's `state.json` contains only that node's data. The engineer's root `state.json` is a centralized index of the full tree structure for fast navigation.

```
.wolfcastle/system/projects/wild-macbook/
  state.json                          # Root index: full tree structure registry
  attunement-tree/
    state.json                        # Orchestrator node state
    attunement-tree.md                # Project description (model-written)
    fire-impl/
      state.json                      # Leaf node state
      fire-impl.md                    # Project description (model-written)
      task-3.md                       # Task working doc (model-created, optional)
    water-impl/
      state.json                      # Leaf node state
      water-impl.md                   # Project description (model-written)
    balance-pass/
      state.json                      # Orchestrator node state
      balance-pass.md                 # Project description (model-written)
      pvp/
        state.json                    # Leaf node state
        pvp.md                        # Project description (model-written)
      pve/
        state.json                    # Leaf node state
        pve.md                        # Project description (model-written)
```

The directory structure under the engineer namespace mirrors the tree address. Every node is a directory containing its own `state.json` and an optional Markdown description file named `{slug}.md`.

### Root Index: `state.json`

The engineer's root `state.json` sits at the top of the namespace directory:

```
.wolfcastle/system/projects/wild-macbook/state.json
```

This file is a centralized index of the full tree structure (ADR-024). It enables fast navigation and lookup without walking the filesystem. It is Wolfcastle-owned (ADR-002) -- modified only by deterministic scripts, never by the model or user directly.

```json
{
  "version": 1,
  "root_id": "project-root",
  "root_name": "Attunement System",
  "root_state": "in_progress",
  "nodes": {
    "attunement-tree": {
      "name": "Attunement Tree Implementation",
      "type": "orchestrator",
      "state": "in_progress",
      "address": "attunement-tree",
      "decomposition_depth": 0,
      "children": ["attunement-tree/fire-impl", "attunement-tree/water-impl", "attunement-tree/balance-pass"]
    },
    "attunement-tree/fire-impl": {
      "name": "Implement Fire Attunement",
      "type": "leaf",
      "state": "in_progress",
      "address": "attunement-tree/fire-impl",
      "decomposition_depth": 0,
      "parent": "attunement-tree"
    },
    "attunement-tree/water-impl": {
      "name": "Implement Water Attunement",
      "type": "leaf",
      "state": "not_started",
      "address": "attunement-tree/water-impl",
      "decomposition_depth": 0,
      "parent": "attunement-tree"
    },
    "attunement-tree/balance-pass": {
      "name": "Balance Pass",
      "type": "orchestrator",
      "state": "not_started",
      "address": "attunement-tree/balance-pass",
      "decomposition_depth": 0,
      "parent": "attunement-tree",
      "children": ["attunement-tree/balance-pass/pvp", "attunement-tree/balance-pass/pve"]
    },
    "attunement-tree/balance-pass/pvp": {
      "name": "PvP Balance",
      "type": "leaf",
      "state": "not_started",
      "address": "attunement-tree/balance-pass/pvp",
      "decomposition_depth": 0,
      "parent": "attunement-tree/balance-pass"
    },
    "attunement-tree/balance-pass/pve": {
      "name": "PvE Balance",
      "type": "leaf",
      "state": "not_started",
      "address": "attunement-tree/balance-pass/pve",
      "decomposition_depth": 0,
      "parent": "attunement-tree/balance-pass"
    }
  }
}
```

The root index is a flat registry keyed by tree address. Each entry has the node's `name`, `type`, `state`, `address`, `decomposition_depth`, `parent` (absent for top-level children of root), and `children` (present only for orchestrators). Detailed node state (tasks, audit status, failure counters, breadcrumbs) lives in each node's own `state.json`.

### Node State Files

Each node directory contains a `state.json` with that node's operational state. The structure differs by node type.

**Orchestrator node** (`attunement-tree/state.json`):

```json
{
  "version": 1,
  "id": "attunement-tree",
  "name": "Attunement Tree Implementation",
  "type": "orchestrator",
  "state": "in_progress",
  "decomposition_depth": 0,
  "children": [
    { "id": "fire-impl", "address": "attunement-tree/fire-impl", "state": "in_progress" },
    { "id": "water-impl", "address": "attunement-tree/water-impl", "state": "not_started" },
    { "id": "balance-pass", "address": "attunement-tree/balance-pass", "state": "not_started" }
  ],
  "audit": {
    "scope": { "description": "", "files": [], "systems": [], "criteria": [] },
    "breadcrumbs": [],
    "gaps": [],
    "escalations": [],
    "status": "pending",
    "started_at": null,
    "completed_at": null,
    "result_summary": null
  }
}
```

**Leaf node** (`attunement-tree/fire-impl/state.json`):

```json
{
  "version": 1,
  "id": "fire-impl",
  "name": "Implement Fire Attunement",
  "type": "leaf",
  "state": "in_progress",
  "decomposition_depth": 0,
  "tasks": [
    {
      "id": "task-1",
      "description": "Implement base fire attunement spell effects",
      "state": "complete",
      "failure_count": 0
    },
    {
      "id": "task-2",
      "description": "Wire stamina cost to fire spells",
      "state": "in_progress",
      "failure_count": 3
    },
    {
      "id": "task-3",
      "description": "Add fire resistance scaling for PvP balance",
      "state": "not_started",
      "failure_count": 0
    },
    {
      "id": "audit",
      "description": "Verify fire implementation: all spells work, costs applied, tests pass",
      "state": "not_started",
      "failure_count": 0,
      "is_audit": true
    }
  ],
  "audit": {
    "scope": {
      "description": "Verify fire attunement implementation",
      "files": ["combat/attunement.go", "combat/stamina.go"],
      "systems": ["fire-attunement", "stamina-system"],
      "criteria": ["All fire spell tests pass", "Stamina costs applied correctly"]
    },
    "breadcrumbs": [
      {
        "timestamp": "2026-03-12T18:30:00Z",
        "task": "attunement-tree/fire-impl/task-1",
        "text": "Implemented FireBolt and FireWall effects in spell_fire.go"
      }
    ],
    "gaps": [],
    "escalations": [],
    "status": "pending",
    "started_at": null,
    "completed_at": null,
    "result_summary": null
  }
}
```

The audit task is always the last task in a leaf's task list (ADR-007). It is auto-created when the leaf is created and cannot be reordered.

All node `state.json` files are Wolfcastle-owned (ADR-002) -- modified only by deterministic scripts, never by the model or user directly.

### Project Definition Files: Markdown

Each node may have an optional Markdown description file co-located in its directory. The file is named `{slug}.md` where `{slug}` is the node's own slug:

| Address | Definition file path |
|---------|---------------------|
| `attunement-tree` | `.wolfcastle/system/projects/wild-macbook/attunement-tree/attunement-tree.md` |
| `attunement-tree/fire-impl` | `.wolfcastle/system/projects/wild-macbook/attunement-tree/fire-impl/fire-impl.md` |
| `attunement-tree/balance-pass/pvp` | `.wolfcastle/system/projects/wild-macbook/attunement-tree/balance-pass/pvp/pvp.md` |

These Markdown files are model-written documentation (per ADR-010). They describe what the project is about in enough detail for the executing model to understand the scope. They are not state -- they are documentation. The model reads them for context; scripts ignore them.

### Task Descriptions: Hybrid JSON + Markdown

Tasks use a hybrid approach for descriptions (ADR-024):

1. **Brief description** -- a plain-text field in the leaf's `state.json`, set by `wolfcastle task add` and managed by scripts. Always available, always loaded into context.

2. **Rich working document** -- an optional companion Markdown file (e.g. `task-3.md`) co-located in the leaf's directory. The model creates and updates this file during execution with findings, learnings, and context. It is model-written documentation, not state.

Go code controls context injection: the brief JSON description is always included for all tasks in the active leaf; the companion Markdown file is included only for the currently active task to prevent runaway context growth.

Example layout for a leaf with task working documents:

```
.wolfcastle/system/projects/wild-macbook/attunement-tree/fire-impl/
  state.json                      # Leaf state with task list
  fire-impl.md                    # Project description
  task-2.md                       # Working doc for task-2 (model-created)
  task-3.md                       # Working doc for task-3 (model-created)
```

A task without a companion Markdown file is valid. The model decides when a task is complex enough to warrant one.

### No Shared Root Index

Each engineer maintains their own independent tree in their namespace (ADR-024). There is no shared state file across engineers.

- `wolfcastle status` shows only the current engineer's tree (default).
- `wolfcastle status --all` aggregates across all engineer directories at runtime by walking `.wolfcastle/system/projects/*/state.json`.
- No merge conflicts on state files, ever.

When engineer A creates a project and commits the definition files (Markdown), engineer B gets them on pull but they do not automatically appear in B's state. Projects appear in an engineer's state when they start working on them, or `wolfcastle doctor` (ADR-025) can detect orphaned definitions and offer to register them.

## Address Resolution

When the Go code receives a `--node` argument, it resolves it through these steps:

### Step 1: Determine the Engineer Namespace

Read `local/config.json` to get the identity. Concatenate `{user}-{machine}` to produce the namespace. Construct the root index path:

```
.wolfcastle/system/projects/{user}-{machine}/state.json
```

### Step 2: Parse the Root Index

Load and parse the root `state.json`. The `nodes` map is keyed by full tree address. This index provides the tree structure without requiring filesystem walks.

### Step 3: Look Up the Address

The provided `--node` value is used as a direct key lookup in the `nodes` map.

- If the key exists, the node is found. Proceed with the operation.
- If the key does not exist, check whether it is a task address (see Step 4).
- If neither, the address is invalid. Return an error.

### Step 4: Task Address Resolution

If the address has the form `<node-address>/task-N`, split it into the node address and task ID:

- Look up the node address in the root index's `nodes` map.
- Verify the node is a leaf (tasks only exist in leaves).
- Load the node's own `state.json` from the corresponding directory.
- Search the node's `tasks` array for a task with the matching `id`.
- If found, the task is resolved. If not, the address is invalid.

### Step 5: Load Node State (if needed)

For operations that need the full node state (task list, audit status, failure counters), construct the node's `state.json` path by converting the tree address to a directory path:

```
.wolfcastle/system/projects/{user}-{machine}/{address-with-slashes-as-directories}/state.json
```

For example, `attunement-tree/fire-impl` resolves to:

```
.wolfcastle/system/projects/wild-macbook/attunement-tree/fire-impl/state.json
```

### Step 6: Resolve the Definition File (if needed)

For operations that need the project description (e.g., building the execution prompt), construct the Markdown file path. The definition file is co-located in the node's directory, named after the node's own slug:

```
.wolfcastle/system/projects/{user}-{machine}/{address-as-directory}/{slug}.md
```

For example, `attunement-tree/fire-impl` resolves to:

```
.wolfcastle/system/projects/wild-macbook/attunement-tree/fire-impl/fire-impl.md
```

If the file exists, read it. If it does not exist, the node has no description (this is valid -- descriptions are optional).

### Step 7: Resolve Task Working Document (if needed)

For the active task during execution, construct the companion Markdown path:

```
.wolfcastle/system/projects/{user}-{machine}/{leaf-address-as-directory}/{task-id}.md
```

For example, `task-3` in `attunement-tree/fire-impl` resolves to:

```
.wolfcastle/system/projects/wild-macbook/attunement-tree/fire-impl/task-3.md
```

If the file exists, include it in the execution context. If not, the task has no working document (this is valid -- companion documents are optional and model-created).

## Navigation: Depth-First Traversal

`wolfcastle navigate` finds the next actionable leaf by performing a depth-first traversal of the tree (ADR-014). It can be scoped to a subtree via `--node`.

### Algorithm

Navigation uses the root index for tree structure and loads individual node `state.json` files only when needed to check actionability:

```
navigate(root_address):
  node_entry = root_index.nodes[root_address]

  if node_entry.type == "leaf":
    node_state = load_node_state(root_address)
    if has_actionable_task(node_state):
      return root_address
    else:
      return nil  # leaf is fully complete or fully blocked

  # node is an orchestrator -- visit children depth-first
  for child_slug in node_entry.children:
    child_address = root_address + "/" + child_slug
    result = navigate(child_address)
    if result != nil:
      return result

  # all children exhausted -- check orchestrator audit
  node_state = load_node_state(root_address)
  if node_state.audit.status == "pending" and all_children_complete(node_entry):
    return root_address  # orchestrator itself needs audit

  return nil  # entire subtree is done
```

The root index provides the tree structure (types and children lists) so that navigation can traverse the tree without loading every node's state file. Individual `state.json` files are loaded only for nodes that need actionability checks.

### What Makes a Leaf Actionable

A leaf is actionable if it has at least one task with state `not_started` or `in_progress` that is not blocked. Tasks are processed in order -- the first non-complete, non-blocked task is the active task.

### Scoped Navigation

When `wolfcastle start --node attunement-tree/balance-pass` is invoked, navigation is rooted at `attunement-tree/balance-pass` instead of the tree root. The daemon only visits nodes under that subtree. Addresses outside the scope are invisible to the navigation loop.

### Traversal Order

Children of an orchestrator are visited in the order they appear in the `children` array (both in the root index and in the orchestrator's own `state.json` -- these must agree). This order is set when children are created (via `wolfcastle project create`) and is stable. The first child is fully explored (depth-first) before the second child is visited.

For the example tree, the full traversal order is:

1. `attunement-tree/fire-impl` (all tasks)
2. `attunement-tree/water-impl` (all tasks)
3. `attunement-tree/balance-pass/pvp` (all tasks)
4. `attunement-tree/balance-pass/pve` (all tasks)
5. `attunement-tree/balance-pass` (orchestrator audit)
6. `attunement-tree` (root orchestrator audit)

## Address Validation

### Valid Addresses

An address is valid if:

1. It conforms to the grammar (slash-separated kebab-case slugs).
2. It exists as a key in the root index's `nodes` map, OR it is a task address (`<node-address>/task-N`) where the node exists and contains a task with that ID.

### Invalid Addresses

An address is invalid if any of the following are true:

| Condition | Example | Error |
|-----------|---------|-------|
| Empty string | `""` | "address must not be empty" |
| Leading slash | `/attunement-tree` | "address must not start with /" |
| Trailing slash | `attunement-tree/` | "address must not end with /" |
| Consecutive slashes | `attunement-tree//fire-impl` | "address must not contain empty segments" |
| Invalid slug characters | `attunement_tree` | "slug contains invalid characters: use lowercase letters, digits, and hyphens" |
| Slug starts with digit | `3fire-impl` | "slug must start with a letter" |
| Slug starts with hyphen | `-fire-impl` | "slug must start with a letter" |
| Slug ends with hyphen | `fire-impl-` | "slug must not end with a hyphen" |
| Node not found | `attunement-tree/ice-impl` | "node not found: attunement-tree/ice-impl" |
| Task not found | `attunement-tree/fire-impl/task-99` | "task not found: task-99 in attunement-tree/fire-impl" |
| Task on orchestrator | `attunement-tree/task-1` | "cannot address task on orchestrator node" |

Validation happens in two phases: syntactic validation (format checks, no state needed) and semantic validation (existence checks against the root index and node state files). Syntactic validation runs first so that malformed addresses produce clear errors without touching the filesystem.

## Creating Nodes and Their Addresses

### Creating a Project

```
wolfcastle project create "attunement-tree"
```

When `--node` is omitted, the project is created at the root level. The slug is derived from the name argument by lowercasing and replacing spaces/special characters with hyphens.

This performs three filesystem operations:

1. Adds the node to the root index (`state.json`) `nodes` map:

```json
{
  "version": 1,
  "root_id": "project-root",
  "root_name": "My Project",
  "root_state": "not_started",
  "nodes": {
    "attunement-tree": {
      "name": "Attunement Tree",
      "type": "orchestrator",
      "state": "not_started",
      "address": "attunement-tree",
      "decomposition_depth": 0,
      "children": []
    }
  }
}
```

2. Creates the node directory and its `state.json`:

```
.wolfcastle/system/projects/wild-macbook/attunement-tree/state.json
```

```json
{
  "version": 1,
  "id": "attunement-tree",
  "name": "Attunement Tree",
  "type": "orchestrator",
  "state": "not_started",
  "decomposition_depth": 0,
  "children": [],
  "audit": {
    "scope": { "description": "", "files": [], "systems": [], "criteria": [] },
    "breadcrumbs": [],
    "gaps": [],
    "escalations": [],
    "status": "pending",
    "started_at": null,
    "completed_at": null,
    "result_summary": null
  }
}
```

3. Optionally, the model creates the definition file at:

```
.wolfcastle/system/projects/wild-macbook/attunement-tree/attunement-tree.md
```

### Creating a Sub-Project

```
wolfcastle project create --node attunement-tree "fire-impl"
```

This validates that `attunement-tree` exists and is an orchestrator, then:

1. Adds `fire-impl` to `attunement-tree`'s `children` array in both the root index and the orchestrator's own `state.json`.
2. Creates the root index `nodes` entry keyed as `attunement-tree/fire-impl`.
3. Creates the node directory `.wolfcastle/system/projects/wild-macbook/attunement-tree/fire-impl/` with its own `state.json`.
4. The new node's type (orchestrator or leaf) defaults to leaf. An orchestrator is created by subsequently adding children to it.

### Creating Tasks

```
wolfcastle task add --node attunement-tree/fire-impl "Wire stamina cost to fire spells"
```

This validates that `attunement-tree/fire-impl` is a leaf node, then:

1. Determines the next task ID (`task-N` where N is one greater than the current highest).
2. Inserts the new task before the audit task (audit is always last) in the leaf's `state.json`.
3. Returns the full task address: `attunement-tree/fire-impl/task-2`.

The brief description is stored in the leaf's `state.json`. The model may later create a companion working document at `attunement-tree/fire-impl/task-2.md` during execution.

The returned address can be used in subsequent commands:

```
wolfcastle task claim --node attunement-tree/fire-impl/task-2
```

### Converting a Leaf to an Orchestrator

When `wolfcastle project create` is called on a node that is currently a leaf, the node is promoted to an orchestrator. This is only allowed if the leaf has no tasks (or all tasks have been archived). Existing tasks prevent promotion -- they must be completed and archived first, or the user must explicitly restructure the tree.

Promotion updates both the root index entry and the node's own `state.json` to reflect the type change.

## Consistency Between Root Index and Node State Files

The root index and individual node `state.json` files contain overlapping information (node type, children lists). These must remain consistent. The rules are:

1. **Scripts update both atomically.** Any command that modifies tree structure (create, promote, archive) updates the root index and the affected node state files in the same operation.
2. **Root index is authoritative for navigation.** The navigation algorithm reads only the root index for tree traversal. If a discrepancy exists, navigation follows the root index.
3. **Node state is authoritative for operational data.** Task lists, audit status, failure counters, and breadcrumbs live only in node state files. The root index never duplicates this data.
4. **`wolfcastle doctor` detects and repairs drift.** If the root index and node state files disagree (e.g., after a failed partial write or manual edit), `wolfcastle doctor` (ADR-025) identifies the inconsistency and offers to repair it.

## Relationship Between Definition Files, Task Documents, and State Files

Definition files (Markdown), task working documents (Markdown), and state files (JSON) serve fundamentally different purposes and are governed by different rules:

| Aspect | Definition Files ({slug}.md) | Task Working Docs (task-N.md) | Root Index (root state.json) | Node State Files (node state.json) |
|--------|------------------------------|-------------------------------|------------------------------|-----------------------------------|
| **Purpose** | Describe what a project is about | Rich findings, learnings, context for a task | Tree structure registry for fast navigation | Track node-level state and task status |
| **Written by** | The model | The model | Deterministic scripts only | Deterministic scripts only |
| **Read by** | The model (for context) | The model (active task only) | Scripts and the Go binary | Scripts and the Go binary |
| **Modified by** | The model (directly) | The model (directly) | `wolfcastle` commands only | `wolfcastle` commands only |
| **Required** | No (optional per node) | No (model decides) | Yes (one per engineer namespace) | Yes (one per node) |
| **Location** | Co-located in node directory | Co-located in leaf directory | Engineer namespace root | Co-located in node directory |
| **Content** | Prose: goals, scope, constraints | Prose: findings, decisions, working notes | Structured: names, types, states, addresses, children | Structured: tasks, states, counters, audit |

The model reads definition files and task working documents to understand what it should build and what it has learned. It calls scripts to record progress in state files. They are connected by the tree address, which maps to a directory containing all three.
