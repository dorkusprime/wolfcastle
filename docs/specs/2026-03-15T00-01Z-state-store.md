# StateStore Design

This spec defines the `StateStore` abstraction that replaces raw Load/Save pairs as the primary API for all file-backed state mutations in Wolfcastle. It governs ADR-068 (Unified StateStore for File-Backed State) and builds on ADR-042 (State File Locking) and ADR-024 (Distributed State Files).

## Governing ADRs

- ADR-024: Distributed State Files, Task Working Documents, and Runtime Aggregation
- ADR-042: State File Locking
- ADR-068: Unified StateStore for File-Backed State

---

## 1. Interface

```go
type StateStore struct {
    dir  string    // namespace/projects directory
    lock *FileLock // per-namespace advisory lock
}

// NewStateStore creates a store for the given namespace directory.
func NewStateStore(namespaceDir string) *StateStore

// Read operations (no lock needed — atomic writes guarantee complete reads)
func (s *StateStore) ReadNode(addr string) (*NodeState, error)
func (s *StateStore) ReadIndex() (*RootIndex, error)
func (s *StateStore) ReadInbox() (*InboxFile, error)

// Mutation operations (lock → read → apply → atomic write → unlock)
func (s *StateStore) MutateNode(addr string, fn func(*NodeState) error) error
func (s *StateStore) MutateIndex(fn func(*RootIndex) error) error
func (s *StateStore) MutateInbox(fn func(*InboxFile) error) error
```

The `fn` callback receives the current on-disk state. It modifies the struct in place. If `fn` returns nil, the modified struct is written atomically. If `fn` returns an error, no write occurs and that error propagates to the caller.

---

## 2. Path Resolution

The store resolves node addresses to file paths internally, using `tree.ParseAddress` to split the address into directory components.

```go
func (s *StateStore) nodePath(addr string) (string, error) {
    a, err := tree.ParseAddress(addr)
    if err != nil {
        return "", err
    }
    return filepath.Join(s.dir, filepath.Join(a.Parts...), "state.json"), nil
}

func (s *StateStore) indexPath() string {
    return filepath.Join(s.dir, "state.json")
}

func (s *StateStore) inboxPath() string {
    return filepath.Join(s.dir, "inbox.json")
}
```

The store accepts the projects directory (from the resolver) as its `dir` parameter. Callers no longer need to assemble file paths from addresses; the store encapsulates that mapping entirely.

---

## 3. Locking Strategy

One lock per namespace, stored as `.lock` in the namespace directory. This matches the existing `FileLock` granularity from ADR-042.

**Mutate methods** acquire the lock for the full duration of read + callback + write. The sequence:

1. `lock.Acquire()` (blocks up to timeout, default 5s)
2. Read the target file from disk
3. Invoke the callback
4. If the callback returned nil, `atomicWriteJSON` the result
5. `lock.Release()` (via defer, so it runs even on panic)

**Read methods** skip the lock entirely. Because `atomicWriteJSON` writes to a temp file then renames (an atomic operation on POSIX and Windows for same-filesystem renames), a concurrent reader either sees the old file or the new file. No torn reads are possible.

**Lock timeout** uses `DefaultLockTimeout` (5 seconds). If the lock cannot be acquired, `ErrLockTimeout` propagates to the caller with a message indicating the daemon is mid-iteration.

**Stale lock detection** is already handled by `FileLock.tryCleanStaleLock`, which reads the PID from the lock file and checks whether that process is still alive.

---

## 4. Atomic Writes

Already implemented in `atomicWriteJSON`:

1. Marshal the struct to indented JSON
2. Write to a temp file in the same directory (`.wolfcastle-tmp-*`)
3. Close the temp file
4. `os.Rename` the temp file to the target path

The rename is atomic on POSIX (guaranteed by the filesystem when source and destination are on the same mount). On Windows, `os.Rename` calls `MoveFileEx` with `MOVEFILE_REPLACE_EXISTING`, which is atomic for same-volume moves. Because the temp file is created in the same directory as the target, the same-filesystem constraint is always satisfied.

## 5. How propagateState Changes

**Current behavior:** `propagateState` accepts an `*RootIndex` that was loaded at the start of the iteration. Before writing, it defensively re-reads the index from disk to pick up nodes created by CLI commands the intake model called mid-iteration. It then copies the fresh index back into the caller's pointer so subsequent operations in the same iteration see the updates.

This is the re-read workaround. It mitigates the clobber race but doesn't eliminate it: there's still a window between the re-read and the write where another process could modify the index.

**After StateStore:** Each state transition calls `store.MutateIndex(func(idx *RootIndex) error { ... })`. The callback receives the index as it exists on disk at the moment the lock is held. There is no window for concurrent modification because the lock serializes all mutations. The re-read workaround, the fallback to the in-memory copy, and the pointer-copy dance all become unnecessary.

The propagation function changes from:

```
propagateState(nodeAddr, nodeState, idx) → reads idx from disk, merges, writes
```

to:

```
store.MutateIndex(func(idx) {
    idx.Nodes[nodeAddr].State = nodeState
    propagate ancestors...
})
```

The `Propagate` function itself (which walks ancestors and updates their state) runs inside the callback, operating on the freshly-loaded index.

## 6. How the Daemon Uses It

The `Daemon` struct gains a `Store *StateStore` field, constructed from `d.Resolver.ProjectsDir()` during daemon initialization.

### runIteration

| Current pattern | StateStore pattern |
|---|---|
| `LoadNodeState(statePath)` then `SaveNodeState(statePath, ns)` | `store.MutateNode(addr, func(ns) { TaskClaim(ns, taskID); ... })` |
| Load, modify, save for task block | `store.MutateNode(addr, func(ns) { TaskBlock(ns, taskID, reason) })` |
| Load, modify, save for task complete | `store.MutateNode(addr, func(ns) { TaskComplete(ns, taskID) })` |
| Load, modify, save for failure increment | `store.MutateNode(addr, func(ns) { IncrementFailure(ns, taskID) })` |

The post-invocation reload (`state.LoadNodeState` after model execution) remains: the model may have called CLI commands that mutated state outside the daemon's lock scope. The reload itself becomes `store.ReadNode(addr)`, which is a lockless read.

### propagateState

Replaced entirely by `store.MutateIndex` as described in section 5. The `propagateState` method either becomes a thin wrapper around `store.MutateIndex` or is inlined at call sites.

### runIntakeStage (inbox goroutine)

Currently uses `InboxMutate` directly. After migration, the inbox goroutine uses `store.MutateInbox` instead. Because both methods serialize through the same namespace lock, the behavior is identical; only the entry point changes.

## 7. How CLI Commands Use It

CLI commands construct the store from the resolver's namespace directory:

```go
store := state.NewStateStore(app.Resolver.ProjectsDir())
```

### cmd/task/add.go

Currently: `LoadNodeState` → `TaskAdd` → `SaveNodeState` (no lock).

After: `store.MutateNode(nodeAddr, func(ns *NodeState) error { TaskAdd(ns, title); ... })`. The mutation is atomic. If the daemon is mid-iteration and holding the lock, the CLI command waits up to 5 seconds, then fails with a clear timeout message.

### cmd/project/create.go

Currently: `LoadRootIndex` → modify → `SaveRootIndex` (no lock). Also loads and saves parent node state without locking.

After: `store.MutateIndex(func(idx) { ... })` for the root index update, and `store.MutateNode(parentAddr, func(ns) { ... })` for the parent node promotion and child ref addition.

This is the most impactful migration. The create command currently does multiple unlocked read-modify-write cycles (root index, parent node state, new node state). With the store, each mutation is independently atomic. The root index update and the parent node update are separate `Mutate*` calls, each holding the lock for its own read-callback-write cycle. This is sufficient because the operations are independent: the root index entry and the parent's child ref don't need to be written in a single transaction.

### cmd/audit/breadcrumb.go

Currently: `LoadNodeState` → `AddBreadcrumb` → `SaveNodeState` (no lock).

After: `store.MutateNode(nodeAddr, func(ns) { AddBreadcrumb(ns, nodeAddr, text, clock) })`.

### Other audit commands (scope, gap, approve)

Same pattern. Any CLI command that calls `LoadNodeState` then `SaveNodeState` migrates to `store.MutateNode`.

---

## 8. Concurrency Model

The lock serializes all mutations within a single namespace. This means:

- Two CLI commands targeting the same namespace execute their mutations sequentially.
- The daemon and a CLI command targeting the same namespace serialize their mutations.
- Two daemons (which ADR-020's PID check prevents, but defensively) would serialize rather than corrupt.
- Multiple namespaces can mutate concurrently, each with their own lock file.
- Read operations are always safe regardless of concurrent writes (no lock, no stale cache).

The daemon and CLI processes coordinate exclusively through the filesystem lock. No shared memory, no IPC, no sockets.

---

## 9. Error Handling

| Condition | Behavior |
|---|---|
| Callback returns an error | Mutation aborted. No write. Error propagated to caller. Lock released. |
| `atomicWriteJSON` fails (disk full, permission denied) | Lock released. Error propagated. On-disk state is unchanged (the rename never happened). |
| Lock times out (5s default) | `ErrLockTimeout` returned. No read, no callback, no write. |
| Lock file contains stale PID (dead process) | `FileLock.tryCleanStaleLock` detects and cleans it automatically. Next acquisition attempt succeeds. |
| State file doesn't exist (first write) | `ReadNode` and `ReadIndex` return an error. `ReadInbox` returns an empty `InboxFile`. For node and index creation, the caller uses `SaveNodeState`/`SaveRootIndex` directly (creation is not a read-modify-write; there's nothing to read). |

## 10. Migration Strategy

### Phase 1: Build StateStore

Add the `StateStore` struct and its methods to `internal/state/`. Unit tests cover the lock-read-callback-write sequence, error propagation on callback failure, and concurrent mutation safety.

### Phase 2: Migrate daemon callers

Wire `Store` into `Daemon`. Migrate `propagateState` to `store.MutateIndex`. Migrate `runIteration`'s task claim, complete, block, and failure paths to `store.MutateNode`. Migrate the inbox goroutine to `store.MutateInbox`.

### Phase 3: Migrate CLI callers

Migrate `cmd/task/add.go`, `cmd/project/create.go`, `cmd/audit/breadcrumb.go`, and all other CLI commands that do Load/Save pairs. Each command constructs the store from `app.Resolver.ProjectsDir()`.

### Phase 4: Deprecate raw Load/Save functions

Mark `LoadNodeState`/`SaveNodeState`, `LoadRootIndex`/`SaveRootIndex`, and the standalone `InboxMutate`/`InboxAppend` as deprecated. Keep them for tests and any edge cases that genuinely need raw access.

### Phase 5: Remove the propagateState re-read workaround

The defensive re-read, the in-memory fallback, and the pointer-copy dance in `propagateState` are no longer needed. Remove them.

---

## 11. What NOT to Build

- **No write-ahead log.** Wolfcastle's write volume is a handful of state transitions per daemon iteration, measured in single-digit seconds. A WAL would add complexity for no measurable benefit.
- **No multi-file transactions.** Each `Mutate*` call is independent. The root index and a node's state file are never written in a single atomic operation. This is acceptable because the operations are logically independent: the index entry and the node state file can briefly disagree, and the next mutation or `doctor` run resolves any inconsistency.
- **No in-memory caching.** The store always reads from disk inside the lock. Caching would introduce stale-read bugs, the exact problem the store is designed to eliminate.
- **No schema versioning beyond what exists.** The `version` field in state files is already present and sufficient. The store does not introduce migration logic or version negotiation.

## 12. Relationship to Existing Code

| Existing code | After StateStore |
|---|---|
| `atomicWriteJSON` | Unchanged. Used internally by all `Mutate*` methods. |
| `FileLock` | Unchanged. The store holds one and delegates to it. |
| `InboxMutate` / `InboxAppend` | Become thin wrappers that construct a `StateStore` and call `MutateInbox`. Eventually deprecated. |
| `LoadNodeState` / `SaveNodeState` | Kept for tests and raw access. Production callers migrate to `MutateNode`. |
| `LoadRootIndex` / `SaveRootIndex` | Kept for tests and raw access. Production callers migrate to `MutateIndex`. |
| `propagateState` re-read workaround | Removed in Phase 5. |
| `runIteration` post-model reload | Changes from `LoadNodeState(path)` to `store.ReadNode(addr)`. Still needed because the model may have mutated state via CLI commands outside the lock. |
