# Auto-Archive Service Contract

## Overview

The auto-archive service moves completed top-level project trees from active state to archive storage after a configurable delay. It preserves the existing Markdown rollup format (ADR-016) while relocating structural state files out of the active index, keeping the daemon's working set lean as projects accumulate.

This spec defines six contracts: the archive state model, file layout, operations, config schema, daemon timer interface, and status integration.

## 1. Archive State Model

### Completion Timestamp Tracking

The `AuditState` struct already carries `CompletedAt *time.Time`. This field is set when a node's audit completes (status transitions to `passed` or `failed`). For archive eligibility, the daemon uses the `CompletedAt` timestamp on the **root-level orchestrator's** audit state. No new fields on `NodeState` are needed.

A new field on `IndexEntry` tracks archived nodes:

```go
type IndexEntry struct {
    // ... existing fields ...
    Archived   bool       `json:"archived,omitempty"`
    ArchivedAt *time.Time `json:"archived_at,omitempty"`
}
```

### What "Archived" Means

Archiving is **not** a new state in the node lifecycle state machine (`not_started`, `in_progress`, `complete`, `blocked`). The four-state model defined in the state machine spec remains unchanged. Instead, archiving is an orthogonal flag on `IndexEntry`:

- The node retains `State: "complete"` in both its `IndexEntry` and its `NodeState`.
- `Archived: true` and `ArchivedAt` are set on the `IndexEntry`.
- The node's state directory is relocated from `system/projects/{namespace}/{node-path}/` to `system/projects/{namespace}/.archive/{node-path}/`.
- The `RootIndex.Nodes` map retains the entry (for status queries) but the entry is flagged as archived.
- Active tree navigation (`FindNextTask`) skips entries where `Archived == true`.

This preserves the state machine's integrity: a "complete" node is complete whether or not it has been archived. Archiving is a storage-layer concern, invisible to the lifecycle.

### Root Array Management

When a root-level node is archived, its address is moved from `RootIndex.Root` to a new `RootIndex.ArchivedRoot` slice:

```go
type RootIndex struct {
    // ... existing fields ...
    ArchivedRoot []string `json:"archived_root,omitempty"`
}
```

This keeps `Root` as the hot path for navigation (no need to filter archived entries on every `FindNextTask` call) while preserving the archived addresses for status and restore.

## 2. File Layout

### Active State (unchanged)

```
.wolfcastle/system/projects/{namespace}/
├── state.json                    # RootIndex
├── {node-address}/
│   └── state.json                # NodeState
└── {node-address}/{child}/
    └── state.json
```

### Archived State

```
.wolfcastle/system/projects/{namespace}/.archive/
├── {node-address}/
│   └── state.json                # NodeState (preserved as-is)
└── {node-address}/{child}/
    └── state.json
```

The `.archive/` directory lives inside the namespace directory, keeping archived state co-located with its namespace. The leading dot signals "not part of active traversal" to both the daemon and human readers.

The directory hierarchy is **preserved**, not flattened. A node at `my-project/auth/login` archives to `.archive/my-project/auth/login/state.json`. This makes restore a symmetric directory move and preserves the original structure for debugging.

### Markdown Rollup Entries (unchanged)

The existing `.wolfcastle/archive/` directory at the project root continues to hold Markdown rollup entries generated by `archive.GenerateEntry()`. These are human-readable summaries, distinct from the structural state files in `.archive/`. The Markdown entries are never moved or deleted by the auto-archive service; they are append-only records.

## 3. Operations

### Move (Archive)

**Preconditions:**
- Node must be a root-level entry in `RootIndex.Root`.
- Node's `IndexEntry.State` must be `complete`.
- Node must not already be archived (`Archived == false` or absent).
- All descendant nodes must also be `complete` (validated by walking `Children` in the index).

**Procedure:**
1. Generate the Markdown rollup entry via `archive.GenerateEntry()` and write it to `.wolfcastle/archive/`.
2. For each node in the subtree (root and all descendants, depth-first):
   a. Move `system/projects/{namespace}/{addr}/` to `system/projects/{namespace}/.archive/{addr}/`.
3. Update the `RootIndex`:
   a. Remove the root address from `Root` and append to `ArchivedRoot`.
   b. For each node in the subtree, set `Archived: true` and `ArchivedAt: now` on its `IndexEntry`.
4. Write the updated `RootIndex` to disk.

**Error handling:**
- If the directory move fails partway, the operation is not atomic. The daemon must detect this on the next poll: if `.archive/{addr}/` exists but `Archived` is false in the index, it resumes from step 3 (idempotent index update). If the active directory still exists alongside the archive copy, the active copy is authoritative and the archive copy is removed (retry on next poll).
- If the node's state changed between eligibility check and move (e.g., a restore happened, or new tasks were added), abort silently. The next poll will re-evaluate.

### Restore (Un-archive)

**Preconditions:**
- Node must exist in `RootIndex.Nodes` with `Archived == true`.
- Node must be a root-level entry in `ArchivedRoot`.

**Procedure:**
1. For each node in the subtree (depth-first):
   a. Move `system/projects/{namespace}/.archive/{addr}/` back to `system/projects/{namespace}/{addr}/`.
2. Update the `RootIndex`:
   a. Remove from `ArchivedRoot`, append to `Root`.
   b. Clear `Archived` and `ArchivedAt` on all subtree `IndexEntry` records.
3. Write the updated `RootIndex`.

**Post-restore state:** The node returns in its original `complete` state. The caller may transition it to `not_started` or `in_progress` manually if new work is needed, but the restore operation itself makes no state machine transitions. This avoids surprising side effects.

**CLI:** `wolfcastle archive restore --node {addr}`

### Delete (Permanent Removal)

**Preconditions:**
- Node must exist in `RootIndex.Nodes` with `Archived == true`.
- Node must be a root-level entry in `ArchivedRoot`.

**Procedure:**
1. Remove `system/projects/{namespace}/.archive/{addr}/` and all descendant directories.
2. Remove all subtree entries from `RootIndex.Nodes`.
3. Remove the root address from `ArchivedRoot`.
4. Write the updated `RootIndex`.

Markdown rollup entries in `.wolfcastle/archive/` are **not** deleted. They are permanent records.

**CLI:** `wolfcastle archive delete --node {addr}` (requires `--confirm` flag to prevent accidents).

## 4. Config Schema

```go
type ArchiveConfig struct {
    AutoArchiveEnabled    bool `json:"auto_archive_enabled"`
    AutoArchiveDelayHours int  `json:"auto_archive_delay_hours"`
    PollIntervalSeconds   int  `json:"archive_poll_interval_seconds"`
}
```

**Defaults:**

| Field | Default | Rationale |
|-------|---------|-----------|
| `AutoArchiveEnabled` | `true` | Archive by default; users disable if they want completed trees visible indefinitely. |
| `AutoArchiveDelayHours` | `24` | One full day after completion, giving the human time to review results before archival. |
| `PollIntervalSeconds` | `300` | Archive checks are infrequent (5 minutes). No urgency in archival; this keeps the daemon's hot path uncluttered. |

**Location in Config tree:**

```go
type Config struct {
    // ... existing fields ...
    Archive ArchiveConfig `json:"archive"`
}
```

The `Archive` field sits at the top level of `Config`, alongside `Daemon`, `Git`, `Summary`, etc. It follows the established pattern: a named sub-struct with JSON tags, registered in the hardcoded defaults function.

## 5. Daemon Timer Interface

### Eligibility Discovery

A node is eligible for auto-archive when all of:
1. It appears in `RootIndex.Root` (top-level only; child nodes archive with their parent).
2. Its `IndexEntry.State == "complete"`.
3. Its `Archived` flag is false or absent.
4. The root node's `AuditState.CompletedAt` is non-nil and `time.Since(CompletedAt) >= AutoArchiveDelayHours`.

The discovery function:

```go
func (d *Daemon) findArchiveEligible(idx *state.RootIndex) []string
```

Iterates `idx.Root`, checks each entry against the criteria above, returns addresses eligible for archival.

### Integration with the Daemon Loop

Archive checking runs **inside RunOnce**, gated behind the "no work" path. When `FindNextTask` returns no actionable work and no planning targets exist, and before entering the idle poll sleep, the daemon checks for archive-eligible nodes:

```
RunOnce:
  1. shutdown / stop-file / iteration-cap checks
  2. branch verification
  3. load index, reconcile orchestrator states
  4. FindNextTask → if found, execute
  5. findPlanningTarget → if found, plan
  6. findArchiveEligible → if found, archive one node, return IterationDidWork
  7. no work → idle poll
```

Archive runs at most **one node per RunOnce call**. This keeps each iteration bounded and allows shutdown checks between archives. If multiple nodes are eligible, they are processed across successive iterations.

This approach requires **no new goroutine**. Archive work is lightweight (directory moves and index updates, no model invocations) and fits naturally into the existing "idle work" slot in RunOnce. The 5-minute `PollIntervalSeconds` in `ArchiveConfig` is enforced by tracking `lastArchiveCheck time.Time` on the Daemon struct and skipping the check if the interval hasn't elapsed.

### Daemon Struct Addition

```go
type Daemon struct {
    // ... existing fields ...
    lastArchiveCheck time.Time
}
```

## 6. Status Integration Contract

### Data Structures

The status command needs two things from the archive system:

**Archive counts** (for the summary line):

```go
type ArchiveSummary struct {
    ArchivedCount int       // len(RootIndex.ArchivedRoot)
    OldestArchive time.Time // earliest ArchivedAt across all archived entries
    NewestArchive time.Time // latest ArchivedAt
}
```

This is derived directly from `RootIndex` by iterating `ArchivedRoot` entries and reading their `ArchivedAt` fields. No separate query interface is needed; the `RootIndex` is the single source of truth.

**Filtered views:**

The status command supports these archive-aware behaviors:

| Flag / Mode | Behavior |
|-------------|----------|
| Default (no flag) | Archived nodes are **excluded** from the tree view. A single summary line appears: `N archived` alongside the existing complete/in_progress/blocked/not_started counts. |
| `--archived` | Show **only** archived nodes in a flat list with address, archived-at timestamp, and original completion date. |
| `--all` (existing) | Includes archived nodes in the tree view, rendered with an `⊘` glyph prefix to distinguish them from active complete nodes. |
| `--json` | Always includes the full `RootIndex` (with archived entries). Consumers filter client-side. |

### Query Interface

No new query interface or service abstraction is needed. The status command already reads the `RootIndex` directly. The archive information lives on `IndexEntry` fields (`Archived`, `ArchivedAt`) and the `ArchivedRoot` slice, all accessible from the same index read the status command already performs.

The rendering logic in `showTreeStatus` adds a filter pass:

```go
// When not showing archived nodes, skip entries where Archived == true.
// When showing only archived, skip entries where Archived == false.
```
